# Failure-triage records

Auto-generated by the future `flashquery-macro-run` skill (Phase 6) when a macro-framework test fails. Each file is `<YYYY-MM-DD>-<HHMMSS>-<test_id>.md` with YAML frontmatter + structured body per §9.6 of the framework requirements.

Records are **committed** (not gitignored) — directed-suite precedent — so PR diffs surface coverage/regression deltas.

## Five classifications (§5.8)

1. **stale-expectations** — auto-detected when the test's `golden_version` is older than the current `macro-golden-model` version. Refresh via `flashquery-macro-testgen --mode=refresh` resolves; no engine / spec change implied.
2. **engine-bug** — production engine deviates from spec; embedded expectations are correct. Fix the engine.
3. **golden-bug** — golden produced wrong expectations at testgen. Patch the golden, bump version, refresh affected tests.
4. **generator-misread** — AI generator translated the spec incorrectly. Fix the generator prompt; regenerate.
5. **spec-ambiguity** — spec is genuinely unclear. File an OQ against the Macro Language Requirements or MCP Broker Requirements; resolution there gates both implementations.

## Lifecycle

`open` -> (`resolved` | `escalated` | `invalidated`). Stale-expectations auto-resolve when refresh succeeds; spec-ambiguity flips to `escalated` once a spec OQ is filed (with `escalated_to:` pointing at the OQ).

Phase 2 does not yet auto-create records — the runner reports assertion failures inline. Phase 6 lands the run skill that writes records here.
