# FlashQuery Core

## What This Is

FlashQuery Core is an open source, local-first data management layer for individual users that sits between their AI tools (any LLM, via MCP) and a unified data store. It manages memory, documents, relational data, and vector embeddings — all the data generated by AI-native workflows — in a system the user owns and controls. It runs locally as a server process, exposes a single MCP server to any AI tool via stdio, stores relational data and vectors in the user's own Supabase instance, and manages a local markdown folder structure that doubles as an Obsidian vault.

## Current State

**Last shipped:** v4.2 JSON Validation (11/11 requirements verified; Phase 170 shipped 2026-06-22).

FlashQuery now has shared LLM JSON repair and schema-validation infrastructure for current LLM-originated and LLM-adjacent parse sites. `parseLlmJson<T>()` provides deterministic repair, typed syntax/schema failures, Zod issue metadata, raw-input retention, and internal repair metadata. Macro evaluator tool results, host-template tool payloads, macro task result envelopes, provider tool-call arguments, and brokered tool text coercion use the shared parser while preserving existing public response envelope conventions and compatibility fallbacks.

Document semantic search continues to run on deterministic markdown chunks stored in `fqc_chunks` rather than whole-document vectors. Documents are parsed into heading-aware chunks with stable UUID5 identity, embedded per active catalog entry, and searched through `match_chunks_<name>` RPCs that aggregate chunk hits into one result row per document with capped nested `matched_chunks`. Memories and plugin records keep the existing row-per-vector embedding model unchanged.

**Current tool surface:** Final primitives center on `write_document`, `write_memory`, `write_record`, `search`, `manage_directory`, `maintain_vault`, and structured read/archive/get flows. Document semantic flows route through the chunk catalog and generated `match_chunks_<name>` RPCs; `maintain_vault` backfill/rebuild operate on chunk rows with by-document reporting.

## Last Milestone: v4.1 Embedding Chunks Migration

**Goal:** Replace whole-document semantic vectors with deterministic markdown chunks stored in `fqc_chunks`; route document semantic search through `match_chunks_<name>` and aggregate to document results; preserve memory/plugin row-per-vector embedding behavior.

**Delivered:**
- Deterministic markdown chunk parser: normalized body hashes (breadcrumb excluded), heading-aware H1–H6 sections, size-guard cascade (paragraph→sentence→token), GFM atomic-block preservation, and stable UUID5 chunk identity.
- Fresh schema stores document semantic data in `fqc_chunks` (per-entry vector column sets + `_indexed_at` stamping) with generated `match_chunks_<name>` RPCs; legacy `fqc_documents.embedding`/`embedding_<name>` columns and `match_documents` RPCs removed.
- Document create/update/copy/scanner/compound flows diff chunks transactionally, delete orphans synchronously, and schedule changed-only `document_chunk` embeddings through the existing pending-embed retry contract.
- Document-scoped `maintain_vault` `backfill_embeddings`/`rebuild_embeddings` operate on chunks with dry-run estimates, by-document reporting, stale filters, and per-chunk failure attribution; memory/record lifecycle preserved.
- Unified document semantic search routes through chunk RPCs, fuses chunk rankings (RRF preserved), and returns one row per document with `limit_chunks_per_result`-capped `matched_chunks` (id, heading path, breadcrumb, content, score, per-entry ranks, freshness).
- Memory and plugin record embedding behavior provably unchanged beside chunked documents; fresh-deployment verification proves chunks are the only document semantic target.

## Context

**Problem:** Every AI tool maintains isolated memory. Claude has its memory, ChatGPT has its memory — switch tools and start from zero. Beyond memory, AI workflows generate notes, decisions, documents, code, and records that live in chat threads with no durable storage.

**Market validation:** Open Brain (600+ GitHub stars in weeks, Supabase-backed memory table via MCP) proves demand. Mem0 ($24M Series A) proves the investment thesis. Neither solves the full problem: Open Brain is one table, Mem0 is memory-only and cloud-SaaS. Nobody provides a unified, local-first data management layer for all AI-generated data types.

**Strategic goals:** Solve the real problem for individual users; gain exposure for FlashQuery brand; attract investment (broader thesis than Mem0's $24M memory-only raise); demonstrate the "dissolving app" thesis; create natural upgrade path to FlashQuery Enterprise.

**Key use case:** "The Dissolved CRM" — contact records in Supabase, contact notes as vault markdown, interaction history appended by AI, browsable in Obsidian and queryable by any AI. No CRM vendor, no lock-in, data owned by the user.

**Current codebase state (v3.6 complete — 2026-05-24):**
- FlashQuery brokers external stdio MCP servers into host, delegated model, and macro surfaces with filtered discovery, schema pinning, tool search/help, diagnostic CLI support, trace/cost evidence, deep health probes, and shared-server safety.
- FlashQuery Macro Language v0 is implemented behind the public `call_macro` MCP tool, including inline/source_ref workflows, `fqm` blocks, control flow, builtins, static permission pre-scan, trace/progress, budgets, and cooperative cancellation.
- Template discovery uses indexed `fqc_documents.template_meta` rows in production and suppresses ordinary non-template warning noise.
- Native `help: true` behavior now has host/delegated parity through a shared dispatch core, while hidden-native and brokered-tool boundaries are preserved.
- v3.6 audit passed 18/18 requirements, 1/1 phases, 18/18 integration, 7/7 flows, and Nyquist compliance.
- 3 CLI commands: `flashquery start`, `flashquery backup`, `flashquery scan`
- Stack: TypeScript strict ESM, Node.js ≥20, `@modelcontextprotocol/sdk@1.27.1`, `@supabase/supabase-js`, `pg`, `gray-matter`, `zod`, `vitest`, `simple-git`, `async-mutex`
- v4.3 source requirements and test plan live outside the repo at `flashquery-product/Roadmap/Features/Graph Document Intelligence (Jun 2026)/Graph-Enhanced-Document-Intelligence Requirements.md` and `Graph-Enhanced-Document-Intelligence Test Plan.md`.
- The milestone builds on shipped v4.1 chunk infrastructure and v4.2 JSON repair infrastructure. Graph nodes use existing `fqc_chunks.id`; Tier 2 uses existing chunk embedding RPCs only for candidate selection; all graph LLM parse sites use `parseLlmJson`.

**Obsidian community angle:** Users who want a "second brain" but stall on maintenance get an auto-organizing vault — the AI does the librarian work, FlashQuery Core does the routing.
