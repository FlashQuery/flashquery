# FlashQuery Core

## What This Is

FlashQuery Core is an open source, local-first data management layer for individual users that sits between their AI tools (any LLM, via MCP) and a unified data store. It manages memory, documents, relational data, and vector embeddings — all the data generated by AI-native workflows — in a system the user owns and controls. It runs locally as a server process, exposes a single MCP server to any AI tool via stdio, stores relational data and vectors in the user's own Supabase instance, and manages a local markdown folder structure that doubles as an Obsidian vault.

**v2.8 shipped (2026-04-21).** Plugin callback overhaul complete: push-based discovery/notification infrastructure replaced with reconcile-on-read pattern. 6 phases, 26 plans, 47 requirements — all satisfied. 1,091 unit / 333 integration / 40 E2E tests passing.

**Phase 90 complete (2026-04-23).** Frontmatter field name centralization complete: all fqc_* and bare-name fields renamed to fq_* equivalents across 8 TypeScript source files, 4 TypeScript test files, and 14 Python test files. FM constants object created as single source of truth. User-defined fields now appear before FQ-managed fields in all vault writes.

## Context

**Problem:** Every AI tool maintains isolated memory. Claude has its memory, ChatGPT has its memory — switch tools and start from zero. Beyond memory, AI workflows generate notes, decisions, documents, code, and records that live in chat threads with no durable storage.

**Market validation:** Open Brain (600+ GitHub stars in weeks, Supabase-backed memory table via MCP) proves demand. Mem0 ($24M Series A) proves the investment thesis. Neither solves the full problem: Open Brain is one table, Mem0 is memory-only and cloud-SaaS. Nobody provides a unified, local-first data management layer for all AI-generated data types.

**Strategic goals:** Solve the real problem for individual users; gain exposure for FlashQuery brand; attract investment (broader thesis than Mem0's $24M memory-only raise); demonstrate the "dissolving app" thesis; create natural upgrade path to FlashQuery Enterprise.

**Key use case:** "The Dissolved CRM" — contact records in Supabase, contact notes as vault markdown, interaction history appended by AI, browsable in Obsidian and queryable by any AI. No CRM vendor, no lock-in, data owned by the user.

## Constraints

- **Tech Stack**: TypeScript + Node.js ESM — `@modelcontextprotocol/sdk@^1.27.1`, `@supabase/supabase-js` + `pg` for DDL, `simple-git`, `js-yaml`, `gray-matter`, `vitest`, `async-mutex`
- **MCP Transport**: stdio only at v1.5 (Claude Desktop/Code spawns FQC as subprocess) — HTTP/SSE deferred
- **Supabase**: Full schema created at v1 startup including v1.5 columns (null defaults) — no migration needed
- **Implementation directory**: All source code in `flashquery-core/src/`, tests in `flashquery-core/tests/`
- **Session continuity**: Each phase must update `flashquery-core/STATUS.md` and task file Completion Log
