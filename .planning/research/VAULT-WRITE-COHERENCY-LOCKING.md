## 1. Summary

FlashQuery's current write/locking strategy applies broad locks around file operations — a design that was practical to implement initially but is becoming a bottleneck as the system evolves toward supporting multiple concurrent processes (agent loops, background scanners, skill runners, and interactive sessions) operating on the same vault simultaneously. The problem is well-understood in systems programming: a coarse lock that covers a large critical section serializes all writers regardless of whether their operations actually conflict, producing unnecessary wait times and making long-running operations (e.g., a large document scan or a bulk tag update) block all other writers for the duration.

The goal of this research is to find the right replacement strategy — one that reduces lock contention without breaking the internal interface layer that all of FlashQuery's consumers depend on. The ideal outcome is a drop-in improvement at the locking layer: callers should not need to know or care about the locking mechanism; they simply acquire a resource and proceed. The new mechanism should support atomic operations, avoid stale-read hazards, and never let a slow long-running operation hold out fast small operations.

Research (2026-05-24) confirms the premise and sharpens it: the current scheme is worse than "coarse" — it is **a single global lock per resource type, routed through a network round-trip to Postgres**, and it does not actually deliver the cross-instance safety its own comments claim. The good news is that the fix is mostly mechanical: the lock infrastructure already supports fine-grained keys (the directory tools already use them), and the atomic-write primitive the document calls for is already in place. See §3 for the full audit; §3.7 holds the staged implementation plan, and §3.11–§3.17 record the decided design.

## 3. Findings

### 3.1 What FlashQuery does today

FlashQuery coordinates writes with **two separate, non-interoperating mechanisms**:

1. **A distributed advisory lock** backed by the Supabase/Postgres table `fqc_write_locks` (introduced in Phase 24). The service is `flashquery/src/services/write-lock.ts`, exposing `acquireLock`, `releaseLock`, and `isLocked`. `acquireLock` deletes expired rows for the resource, then attempts an `INSERT`; a unique-violation means "held," and it retries with exponential backoff (10 ms → 1000 ms cap, default 10 s timeout). Default TTL is 30 s (`flashquery.yml`: `locking.ttl_seconds: 30`). It is used by every write-side MCP tool handler — the document tools (now split under `documents/`), `compound.ts`, `records.ts`, `memory.ts`, `plugins.ts`, `files.ts`.

2. **An in-process mutex** — `scanMutex`, an `async-mutex` `Mutex` (`scanner.ts:28`) — that serializes `runScanOnce()` and `reconcileTrackedDocuments()` against each other.

These two mechanisms do not coordinate. The scanner does **not** acquire the `documents` write lock, and MCP write handlers do not acquire `scanMutex`. So a background scan and an interactive `write_document` are not mutually exclusive with respect to one another, and `repairFrontmatter()` — which writes frontmatter back to disk — runs *outside* `scanMutex` entirely (noted explicitly in a comment at `scanner.ts:1271`). The scanner subsystem and the MCP write subsystem each guard themselves but not each other.

### 3.2 The lock is global per resource type, not per file

Every document-write call site passes the **same literal string** as the lock key:

- `write_document` (`documents/write.ts`) and the sibling document tools `archive_document`, `remove_document`, `copy_document`, `move_document` — each now its own file under `documents/` — → `'documents'`
- `compound.ts` — `insert_in_doc` (`:1062`) and `replace_doc_section` (`:1237`) → `'documents'`; `insert_doc_link` and `apply_tags` take **no lock at all** (the live defect, §3.8)
- `records.ts` → `'records'`; `memory.ts` → `'memory'`; `plugins.ts` → `'plugins'`

So the entire system has roughly four global write locks. **Every document write anywhere in the vault serializes on one `'documents'` lock** — two `write_document` calls to two completely unrelated files cannot proceed in parallel. This is precisely the contention the research request anticipates.

One call site already does better: `files.ts` (the `manage_directory` tool) builds its key as `` `directory:${safePath}` `` (`files.ts:150`). This is the important structural finding — **the lock mechanism already supports arbitrary fine-grained keys**; the `resource_type` column is just a string. The coarseness of the document lock is a *caller choice*, not a limitation of the infrastructure. Moving to per-file locks does not require a new locking subsystem; it requires the document/record/memory callers to construct a better key.

### 3.3 Why the current scheme is a performance hit

Two compounding costs:

**The critical section is large.** In `write_document` (`documents/write.ts`), the `'documents'` lock is acquired at line 66 and released in a `finally` at 312 — it is held across the entire handler body: input and path validation, `existsSync`/`stat`, a file `readFile`, frontmatter serialization, `vaultManager.writeMarkdown` (file write plus a fire-and-forget git commit), a *second* `readFile` to recompute the content hash, one or more Supabase round-trips (`fqc_documents` insert/update, plus `resolveDocumentIdentifier` on the update path), a `targetedScan`, and an awaited `scheduleBackgroundEmbedding`. For `compound.ts` the lock wraps an entire multi-step macro (acquired `:1062`, released `:1199`). So a single slow operation — a large compound macro, an update whose `targetedScan` is expensive — holds the one global lock and blocks every other writer for its full duration.

**The lock itself is a network round-trip.** Because the lock lives in Postgres, *acquiring* it costs a `DELETE` (purge expired rows) plus an `INSERT`, and *releasing* it costs a `DELETE`. Even a completely uncontended write pays roughly two extra Supabase round-trips purely for lock bookkeeping, on top of the operation's own round-trips. Under contention it also pays the backoff-poll loop. For a single local instance — the common case — this is pure overhead for an exclusion problem that is in-process (see §3.7).

The existence of a dedicated `flashquery unlock` CLI command (`cli/commands/unlock.ts`), whose sole job is to clear orphaned `fqc_write_locks` rows, is itself evidence that lock leakage is a real, recurring operational pain point.

A secondary effect: TTL is 30 s. An operation that runs longer than its TTL has its lock silently expire, after which another writer can acquire and proceed *concurrently* — the exact long-running-operation hazard, except it manifests as a silent safety loss rather than a wait.

### 3.4 A correctness defect: the lock does not enforce cross-instance exclusion

`fqc_write_locks` is created (`storage/supabase.ts:521-527`, and again in the migration block `:540-546`) with:

```sql
CREATE TABLE IF NOT EXISTS fqc_write_locks (
  instance_id   TEXT NOT NULL,
  resource_type TEXT NOT NULL,
  locked_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at    TIMESTAMPTZ NOT NULL,
  PRIMARY KEY (instance_id, resource_type)
);
```

The primary key is the **pair** `(instance_id, resource_type)`. Uniqueness is enforced over the pair, so `(instanceA, 'documents')` and `(instanceB, 'documents')` are distinct rows and **both inserts succeed**. The `23505` unique-violation that `acquireLock` treats as "lock held" therefore only ever fires when the *same* instance re-inserts the *same* resource — i.e. the table serializes concurrent async contexts *within one instance* but provides **no mutual exclusion between instances**.

This contradicts the module's own documentation. `write-lock.ts` states "Multiple FQC instances can share one Supabase database; this service ensures only one instance writes a given resource_type at a time" and "PRIMARY KEY (instance_id, resource_type) enforces mutual exclusion." Both claims are false as written. This is tracked as OQ9. The decided design does **not** patch the constraint — it retires the table altogether in favour of native Postgres advisory locks (§3.12), so the defect is removed rather than fixed. It matters to `Research/Multi-Vault.md`, which plans a single shared database across vaults.

### 3.5 Atomic writes — already done; durability is not

The atomic-write primitive the research request asks for **already exists**. `vaultManager.writeMarkdown` (`storage/vault.ts:214-216`) does:

```ts
const tmpPath = absolutePath + '.fqc-tmp';
await writeFile(tmpPath, output, 'utf-8');
await rename(tmpPath, absolutePath);
```

`rename()` within one filesystem is atomic on APFS and on all common Linux filesystems. This is why **no read lock is needed** (see OQ3): a concurrent reader calling `readMarkdown` either opens the old inode or the new one, and always sees a complete, internally consistent file. The "block reads while a write is in progress" goal from the research request is already satisfied structurally — there is simply no torn-read window to protect against.

Three gaps remain:

- **No durability flushing.** A grep for `fsync`/`fdatasync`/`F_FULLFSYNC` across the codebase returns nothing. Atomicity is not durability: after a crash or power loss, the rename may not have reached disk, or the temp file's contents may not have. The correct sequence (per the authoritative write-up in §7) is: write temp file → `fsync` the temp file → `rename` → `fsync` the *containing directory*. The directory fsync is the commonly-missed step — without it the file's contents can survive while the directory entry pointing at it does not. On macOS, durable flushing needs `fcntl(F_FULLFSYNC)`, not plain `fsync`. Node exposes `filehandle.sync()`/`datasync()`; a directory fsync is done by `fs.open(dir)` then `fd.sync()`.
- **Fixed temp name.** The temp path is always `<path>.fqc-tmp`. Two concurrent writers to the *same* document would write to the *same* temp file and corrupt each other. Today nothing at the vault layer prevents this — only the (global) MCP-tool lock does, and only because it serializes *all* document writes. Once locks become per-file this is exactly the case the per-file lock must cover; the temp name should also become unique (include pid + a counter) as defence in depth. `cleanStaleTempFiles()` (`vault.ts:337`) already sweeps stray `.fqc-tmp` files at startup, which is good crash hygiene.
- **Silent write failures.** The frontmatter-only write path `atomicWriteFrontmatter` (`utils/frontmatter.ts`) wraps its whole temp-write-and-rename in a `try/catch` that logs any failure only at `debug` level and then returns normally — the caller is told nothing. A frontmatter write that fails therefore looks like success, leaving the file on disk and the `fqc_documents` row silently divergent. This should be fixed as part of the rework: a failed write must surface as an error, not a swallowed debug log.

### 3.6 Best practices and analogous systems

**Lock granularity.** Fine-grained locks increase concurrency at the cost of per-operation bookkeeping; coarse locks do the reverse. The standard technique to keep fine-grained locking cheap is **lock striping** — a fixed-size pool of N locks with `lock = stripes[hash(key) % N]`. Memory is bounded by N regardless of how many files exist, non-conflicting paths almost always land on different stripes and proceed in parallel, and the only cost is rare, harmless false sharing when two unrelated paths hash to the same stripe. This is how `ConcurrentHashMap` and similar structures get concurrency without a lock per element.

**Readers-writer locks.** An RW lock (many readers OR one writer) is the right tool only when reads are frequent, writes are rare, *and* the critical section is non-trivial. It carries more bookkeeping than a plain mutex, and established guidance ("Reader Locks Should Be Rare") notes it can underperform a plain mutex under contention with short critical sections. For FlashQuery it is simply not needed — atomic rename already decouples readers (§3.5).

**The per-file lockfile pattern (Git).** Git modifies a file by creating `<filename>.lock` with `O_CREAT|O_EXCL` — the exclusive create both *is* the lock and fails fast if another writer holds it — writing into it, and atomically renaming it over the target. The rename commits and unlocks together. Abandoned lockfiles are cleaned by `atexit`/signal handlers. In Node, `proper-lockfile` implements this cross-process: a `<file>.lock` directory, an mtime heartbeat, a configurable stale threshold (default 10 s) so a crashed holder is recoverable, and automatic release on process exit. It is the off-the-shelf model for cross-process file safety *when there is no shared database to lean on*; FlashQuery has one, and §3.9 weighs this filesystem-lockfile approach against the Postgres-native option that was ultimately chosen (§3.12).

**Optimistic concurrency.** Lost updates (OQ4) are solved by versioning, not by locks: capture a version/hash at read time, compare at write time, reject and retry on mismatch (compare-and-swap). FlashQuery already maintains a `content_hash` per document, so it has the version field for free.

**Single-writer stores (SQLite WAL, LMDB).** Both allow unlimited lock-free readers plus exactly one writer, with readers seeing a consistent snapshot. The lesson is not "adopt WAL" — it is that *per-object* single-writer plus *snapshot reads* is a proven, simple shape. FlashQuery's atomic rename already provides the snapshot-read half.

**Node.js concurrency model.** JavaScript in Node runs on a single event-loop thread; there is no true parallel execution of JS. "Concurrency" inside one instance is async operations interleaving across `await` points — which is still enough to create races, but is an *in-process* problem solvable with an in-process mutex (`async-mutex`, already a FlashQuery dependency) at microsecond cost. Cross-*process*/cross-*instance* coordination is the only case that genuinely needs a shared external lock.

### 3.7 Recommended direction

A staged plan, ordered so each step is independently shippable and safety is never reduced:

1. **Make document locks per-file — and close the unlocked tools first.** Change the key from `'documents'` to a per-file key at the `documents/*` and `compound.ts` call sites (the key must canonicalize to the file's *absolute* filesystem path, not a vault-relative path — see OQ21); records, memory, and plugins are database-backed, not file-backed, so per-file keys do not apply to them — their post-`fqc_write_locks` coordination is decided in §3.15 (OQ25). Wrap the document/compound change in a `withDocumentLock(path, fn)` helper — the helper canonicalizes its path argument to the absolute lock key — so every call site is uniform. No change to the `acquireLock`/`releaseLock` signatures (§3.2, OQ8). **Highest priority within this step:** `insert_doc_link` and `apply_tags` (`compound.ts`) currently acquire *no lock at all* (§3.8) — they lose updates outright today — so bringing them under the per-file lock is urgent, not cosmetic.
2. **Shrink the critical section.** Hold the lock only around the read-modify-write of the file itself. The git commit is already fire-and-forget; the awaited `scheduleBackgroundEmbedding` and ideally the `fqc_documents` index update can move *after* lock release (the file is the source of truth; the index trails it and is reconciled by the scanner anyway).
3. **Build the two-tier lock (decided — §3.12).** Tier 1: an in-process striped per-file lock registry on `async-mutex` — microseconds, no network — resolving same-process contention. Tier 2: a global cross-process lock — native Postgres `pg_advisory_lock` on the raw `pg` pool that already exists (`utils/pg-client.ts`), keyed on the file's canonical absolute path. A writer acquires Tier 1, then Tier 2; only one async context per process reaches Tier 2, so the round-trip is paid once per process, not once per caller. Retire `fqc_write_locks` — advisory locks replace it; the records/memory/plugins audit in §3.15 (OQ25) records what each subsystem needs once the coarse lock is gone (memory: nothing; records and plugins: a per-plugin advisory lock or a transaction where a multi-step sequence remains). Full design in §3.12.
4. **Add durability, not read locks.** Do not introduce read locks (§3.5). Instead add `fsync(tempfile)` → `rename` → `fsync(dir)` to `writeMarkdown` (`F_FULLFSYNC` on macOS), and give the temp file a unique name.
5. **Cooperative chunking for bulk passes (OQ6).** Have the scanner, `reconcileTrackedDocuments`, and `repairFrontmatter` acquire/release the per-file lock once per document and yield between documents, rather than running unsynchronized or under one broad lock.
6. **Unify the two mechanisms.** Once per-file locks exist, the scanner's per-document writes and MCP write handlers should acquire the *same* per-file locks, so scans and interactive writes finally coordinate (closes the §3.1 gap).
7. **Close the read-to-write gap — the version-fingerprint check, Layer 0 — decided (OQ11 resolved; §3.10, §3.11).** Surface the content hash on `get_document` and accept an optional `expected_version` / `if_match` precondition on every write tool; reject mismatches with a `conflict` that returns the current content. This converts a silent RWITM lost-update into a clean, detectable rejection, and should ship *with* the rebuild, not after it. Declared modify-intent reservations and merge-on-conflict (OQ18, OQ19) are deferred enhancements — see §3.10.
8. **Define the lock's ordering semantics.** A writer waiting for a contended file is guaranteed a **bounded wait** — it either acquires the lock or fails cleanly within the configured timeout (OQ22; 10 s default), so no writer is starved indefinitely. `async-mutex` (Tier 1) grants waiters in FIFO order *within* a process; the cross-process Tier 2 (`pg_advisory_lock`) does **not** carry a documented FIFO guarantee — Postgres may reorder advisory-lock waiters — so FlashQuery promises bounded-wait fairness, not strict global FIFO (OQ29, §3.16). The current table's exponential-backoff polling delivers neither — a further mark against keeping it. The deliberate *non-*guarantee: between an unsynchronized concurrent read and write of the same file, FlashQuery guarantees *integrity* — the reader sees a complete old or complete new file, never a torn one — but **not ordering**. Which version that read observes is a race; if a caller needs its read ordered against another actor's write, that coordination is the caller's responsibility.

For multi-file atomic operations, acquire the per-file locks in a deterministic order (sorted by path) to prevent deadlock — the standard lock-ordering discipline.

### 3.8 Worst-case stress test: uncoordinated multi-thread access

The §3.7 plan was deliberately stress-tested against the hardest scenario: multiple uncoordinated threads or processes hitting the same vault at once — reading and writing the same file, the same folder, the same frontmatter, linked documents, and performing surgical in-document edits. Each hazard below is tested against the recommendation. The plan holds, with three additions the stress test surfaced.

**Two threads write the same file.** A per-file write lock (§3.7 step 1) serializes them. Crucially, the MCP write and surgical handlers already re-read the file *inside* the lock — verified: `write_document` (`documents/write.ts`) reads at line 215, after `acquireLock` at 66; `insert_in_doc` reads at `compound.ts:1087`, after `acquireLock` at 1062; `replace_doc_section` reads at 1260, after `acquireLock` at 1237. Because the read is inside the lock, the second writer re-reads the file *as the first writer left it* — no lost update. Solved by step 1 with no extra work. The residual hazard is a caller that read via `get_document` earlier and submits a write against that stale view — which is OQ11 (optimistic concurrency), not something a write lock can catch.

**A live defect: two write tools take no lock at all.** `insert_doc_link` (`compound.ts:154`) and `apply_tags` (`compound.ts:280`) perform read-modify-write on document frontmatter with **no `acquireLock` anywhere in their handlers** (verified: no lock calls in lines 154–555; the only locks in `compound.ts` are `insert_in_doc` at 1062 and `replace_doc_section` at 1237). Today, two concurrent `apply_tags` calls on the same document, or an `insert_doc_link` racing a `write_document`, lose updates outright — the second write is computed against a snapshot the first already obsoleted. This is not a future risk; it is live. Bringing these two tools under the per-file lock is the highest-priority part of §3.7 step 1.

**Two threads write the same folder.** Distinct files in one folder, under per-file locks, proceed in parallel — that is the intended win. The real conflict is a *file* write racing a *folder-level* operation: renaming or moving the folder, or a scan/listing that must see a coherent set of files. A `document:${path}` lock and a `directory:${path}` lock are different keys and do not see each other. The textbook fix is **multiple-granularity locking with intention locks** (Gray et al., 1976): before exclusively locking a file, take an *intention-exclusive (IX)* lock on its parent folder; a folder-level operation takes a real shared/exclusive lock on the folder, which conflicts with the children's IX and so waits for in-flight file writes (and blocks new ones). This keeps unlimited file writes parallel while making folder operations safe. FlashQuery already has both key namespaces (`document:` and `directory:`); what was missing is the *protocol* connecting them — now decided (OQ13, resolved): shared/exclusive directory locks, detailed in §3.12.

**Frontmatter of the same file, and linked documents (pointers).** A frontmatter-only write is just a file write — atomic rename replaces the whole file, so there is no "frontmatter vs. body" sub-file race to worry about; per-file locking covers it once `apply_tags`/`insert_doc_link` are locked. The genuinely hard case is **cross-document**:

- *Multi-file writes are not atomic.* `insert_doc_link` mutates N source documents in a loop (`compound.ts:222–238`). Per-file locking makes each individual write safe, but the batch as a whole is not all-or-nothing — a crash or a conflicting concurrent change mid-loop leaves some sources linked and some not. If the operation must be atomic, acquire *all* the per-file locks up front, in sorted canonical-path order (lock ordering prevents deadlock; locking everything before doing any work prevents the partial state), then write under all of them. If best-effort is acceptable, per-file locks plus an idempotent retry suffice. The decided contract — best-effort, with an ordered per-item result envelope — is §3.13 (OQ12, resolved).
- *FlashQuery's pointers use unstable identifiers.* Links are written by **title** (`[[wikilink]]`, built from the target's title at `compound.ts:213`) or by **path** (the `relates_to`/`depends_on` frontmatter — a path that silently rots the moment its target is moved or renamed). If thread A moves or renames the target while thread B links to it, B captures a title or path that is *already wrong*. No lock fixes this, because the identifier itself is unstable. The robust fix is to reference the **stable `fq_id` UUID** FlashQuery already assigns, and resolve to a display title/path at read time — that makes concurrent rename a non-event. This is OQ14. Until then, the scanner should detect and flag dangling pointers as a reconciliation pass.

**Surgical insertions and modifications inside a doc.** `insert_in_doc` and `replace_doc_section` read the whole file, modify a region, and write the whole file back — and (verified above) they read *inside* the lock. So under per-file locking, two surgical edits to two different sections of the *same* file simply serialize: the second re-reads the file already containing the first edit and applies its own change on top. Both edits survive. **FlashQuery does not need Operational Transformation or CRDTs.** Those technologies solve concurrent editing under network latency, offline clients, and long-lived editing sessions (Google-Docs-style). FlashQuery's edits are short, synchronous, and server-mediated; server-side per-file serialization is sufficient and far simpler. The alternatives carry real cost — OT merges are O(n²) or worse, and CRDTs accumulate tombstones (Figma reported documents bloating into the gigabytes). The one rule that must hold: a surgical edit must always apply its change to content it re-read *inside the lock*, never to a snapshot captured earlier. The current tools already do this; the rule must be preserved as an explicit invariant when the lock layer is rebuilt.

**True OS threads (`worker_threads`).** The §3.7 plan's in-process tier (OQ10) is built on `async-mutex`, which coordinates async operations interleaving within *one* event loop. It does **not** span Node `worker_threads`: each worker is a separate V8 isolate with its own heap, so an `async-mutex` instance in one worker is invisible to another. If FlashQuery ever moves vault work onto worker threads, "in-process" locking must become one of: (a) a single dedicated vault-writer thread that all others message — the actor model, which serializes writes by construction and is the simplest; (b) a lock built on `SharedArrayBuffer` + `Atomics.wait`/`notify`; or (c) falling back to the OS file lock or the database lock for cross-thread coordination. Because this choice changes what the in-process tier *is*, the threading model should be decided before the lock layer is rebuilt — OQ15. **Update (2026-05-24): resolved.** Matt has confirmed FlashQuery will not be internally multi-threaded; see §3.9. Within a single process, `async-mutex` is sufficient and this hazard does not arise; coordinating *separate* FlashQuery processes that share a vault is a distinct concern, handled by the global lock (§3.9).

**Net result.** The §3.7 recommendation survived the stress test, and the architecture is now decided (§3.11, §3.12). The threading-model (OQ15), one-process-per-vault (OQ16), and global-lock-implementation (OQ17) questions are all resolved. Two things this stress test surfaced that the rebuild must absorb: (1) `insert_doc_link` and `apply_tags` must be brought under the per-file lock immediately — they are unprotected today; and (2) the file/folder granularity protocol — now decided (OQ13, §3.12: shared/exclusive directory locks). Multi-file atomicity is likewise decided — best-effort, the per-item batch contract in §3.13 (OQ12). Pointer stability (OQ14) is a real improvement but orthogonal to the locking rework and is recommended for a separate item.

### 3.9 The deployment model, and implementing a global (cross-process) lock

**A correction first.** An earlier version of this section claimed that two FlashQuery processes pointed at one vault would "double-track every document," and concluded the architecture implied "one process per vault." *That was wrong, and it is retracted.* It asserted a schema consequence that had not been verified. The verified facts: `fqc_documents` has `PRIMARY KEY (id)` — the document's UUID — with `instance_id` as an ordinary column; and `instance.id` is a stable, operator-set configuration value (`flashquery.yml` → `instance.id`, e.g. `"work-center"`), not something generated per process. So multiple *processes* running with the *same* `instance.id`, pointed at the *same* vault, simply read and write the *same* `fqc_documents` rows — no double-tracking, and an ordinary supported topology. (Two *different* `instance.id`s on one vault is a genuine misconfiguration — they would collide on the `id` primary key — but that is config hygiene, not an architectural law.) **There is no "one process per vault" constraint.**

**The real shape of the problem.** Concurrency reaches FlashQuery in two layers, and both are real:

1. *Intra-process.* Under the default HTTP transport (`mcp.transport: "streamable-http"`, `mcp/server.ts:705`), one FlashQuery process serves many client sessions on one event loop. Many external callers — applications, threads — funnel into that single process. Coordinating them is in-process async interleaving, which an in-process `async-mutex` per-file lock handles at microsecond cost, no network.
2. *Inter-process.* Multiple FlashQuery processes — same `instance.id` or not — may run against the same vault. When two of them write the *same file*, they need mutual exclusion the in-process lock cannot provide, because each process has its own. This is the global lock.

The shared resource in both layers is **the file on disk** — that is what the lock must be keyed on.

**What Postgres already provides — and what FlashQuery uses.** Postgres ships a deep concurrency toolkit: MVCC (readers and writers never block each other); row locks via `SELECT … FOR UPDATE` (pessimistic — the literal "read with intent to modify"); **advisory locks** (`pg_advisory_lock` — a general-purpose named mutex the database provides directly, tied to no row or table); atomic upserts via `INSERT … ON CONFLICT`; and Serializable isolation. FlashQuery already uses several of these — `SELECT … FOR UPDATE` guards the memory-versioning RPC (`supabase.ts:797`), `ON CONFLICT` upserts appear in the embedding and document paths, and the codebase carries a full raw Postgres connection pool (`utils/pg-client.ts`) alongside the Supabase REST client. The one place that uses *none* of it is the write lock: `fqc_write_locks` is a hand-rolled re-implementation of an advisory lock as application data — a row's existence means "held," a unique-violation means "acquire failed," an `expires_at` column is a manual stale-lock TTL, and waiting is a client-side backoff poll. Native `pg_advisory_lock` does all three — exclusion, blocking, and automatic release — natively and correctly.

**Is a global lock a big challenge? No.** The textbook tool — `pg_advisory_lock` — is built into the database FlashQuery already depends on, and the raw connection pool it needs already exists. Three candidate mechanisms were weighed: *native advisory locks* — chosen: correct by construction (no application-defined key to mis-build), native blocking with a bounded wait (OQ29 — not strict FIFO), automatic release when the holding connection drops; *a re-keyed `fqc_write_locks` table* — keeps the hand-rolled TTL and backoff polling and still has to be re-keyed to the file, i.e. strictly more work for a worse result; and *filesystem lockfiles* (`flock` / `proper-lockfile`) — which need no database but are reliable only when every process shares the same *local* filesystem (advisory locks over NFS/SMB are unreliable). An earlier draft of this section claimed advisory locks "do not fit FlashQuery's data access" because the Supabase client is REST-based — that was wrong and is retracted: the raw `pg` pool already exists, and a session-scoped advisory lock held on a checked-out pool connection fits exactly. One genuine subtlety remains: Postgres's locks protect *Postgres rows*, while FlashQuery's contended resource is a *file on disk* — so Postgres serves only as the **rendezvous point** where separate processes meet, not as the thing being locked. Both a table row and an advisory lock fill that rendezvous role; the advisory lock just fills it without the hand-rolled machinery. A resource-keyed lock is also self-scoping: if two instances' vaults touch disjoint files their keys never collide and they never contend; if the vaults overlap, only the overlapping files' keys collide — no "do the vaults overlap?" special-casing is needed.

**Decided — see §3.12.** Tier 2 is native `pg_advisory_lock`, session-scoped, on the existing raw `pg` pool; `fqc_write_locks` is retired; the lock key is the file's canonical absolute path; an in-process `async-mutex` tier sits in front as the fast path. §3.12 states the full decided architecture, with §3.7 the staged plan to get there.

**The one writer the lock will never see.** FlashQuery is not the sole writer of its own vault. The files are Markdown on disk, and external tools — Obsidian, VS Code, `git`, the user's editor — write them directly without calling FlashQuery (`vault.ts` carries an `extractMinimalFrontmatter` path explicitly "for externally-added files"). No lock — in-process or global — can coordinate with Obsidian. That concurrency is handled differently: the scanner's content-hash reconciliation detects external edits after the fact, and the atomic-rename write (§3.5) guarantees even an external reader never sees a torn file. The division of labour: **locks** (in-process + global) coordinate FlashQuery-mediated writes; **reconciliation** absorbs external edits; **atomic rename** protects all readers.

### 3.10 Read with intent to modify: the read-to-write gap

Everything to this point — atomic rename (§3.5), per-file locks (§3.7), the global lock (§3.9) — protects the *moment of writing*. None of it protects the gap *between* a caller's read and that write. This gap is the analogy raised in review to **RWITM (Read With Intent To Modify)** from CPU cache coherency, and it is a real, separate hazard.

**The RWITM analogy, and where it breaks.** In the MESI cache protocol, a core that wants to write a line it holds only shared (or not at all) issues a *Read For Ownership* / RWITM transaction: a single operation that fetches the line *and* invalidates every other core's copy, granting the requester exclusive, modify-capable ownership. The defining property is that **write-intent is declared at read time and fused with acquiring exclusivity** — a plain shared read does not do this. FlashQuery has no equivalent: every `get_document` is a "shared read," and nothing records, signals, or enforces that the caller intends to modify what it read. A second caller can read the same file with the same intent, and both will later write.

The analogy breaks in one decisive way, and the break shapes the whole solution. MESI's RWITM is safe to make *exclusive* because the modify follows the read in nanoseconds — exclusivity is held trivially briefly. FlashQuery's "modify" follows the read after a language model's think-time — seconds to minutes of token generation. Holding a hard exclusive lock across that is exactly what §3.7 warns against: it serializes every other writer behind one caller's latency, and any lock TTL short enough to survive a crash is far shorter than the think-time. So FlashQuery cannot literally copy RWITM — the read-time exclusivity, if any, must be *softer* than a hard lock. (FlashQuery also does not need RWITM's reader-invalidation: atomic rename already gives readers a consistent view — §3.5. The concern is purely writer-vs-writer.)

**How other systems handle the read-to-write gap.** Three established patterns, in increasing cost:

- *Optimistic — ETag / `If-Match` / version check.* The read returns a version token (an ETag); the write submits it back, and the server applies the write only if the resource still matches, else rejects with `412 Precondition Failed`. No coordination at read time; the conflict is caught at write time. This is the cheapest and most widely deployed answer (HTTP, most REST APIs, document databases). Its cost: the conflict surfaces only *after* the caller has done its work — for a language model, after it has spent tokens and latency generating the new content.
- *Pessimistic — `SELECT FOR UPDATE` / WebDAV `LOCK` / document check-out.* The read declares intent and takes an exclusive lock; other intent-readers block. This is the literal RWITM analogue. Its well-known failure is the *forgotten check-out*: a holder that crashes or wanders off leaves the document locked indefinitely — a real, recurring class of bug in document-management systems.
- *Leases — time-bounded locks.* The fix for the forgotten check-out: the lock carries a TTL and auto-expires unless renewed. FlashQuery's existing `fqc_write_locks` already works this way (a 30 s TTL). The residual hazard is the *expiry race* — if the holder's think-time outlives the lease, the lease expires, another writer acquires, and both believe they hold it. The robust guard, per the distributed-systems literature, is a **fencing token**: a monotonically increasing number issued with each lease grant; the write path records the highest token it has seen and rejects any write presenting an older one, so a stale lease-holder's write fails safely even though it never noticed its lease expired.

**What FlashQuery should do — a layered answer.** No single mechanism fits, so the answer is layered — and each layer is now dispositioned: Layer 0 is decided and in scope (§3.11), Layer 1 is dropped and Layer 2 is deferred (both §3.14).

- **Layer 0 — optimistic version check (baseline; should ship with the lock rebuild).** FlashQuery already computes a SHA-256 content hash on read (`document-output.ts:420`) and stores it in `fqc_documents.content_hash` — the version token already exists. Surface it in the `get_document` response, and accept an optional `expected_version` / `if_match` precondition on the write tools (`write_document`, `insert_in_doc`, `replace_doc_section`, `apply_tags`, `insert_doc_link` — and the destructive tools too; §3.15, OQ24). At write time, under the write lock, re-read and compare; on mismatch, reject with a `conflict` error **that returns the current content and hash** so the caller can re-read and retry. This does not prevent the race — it converts it from a *silent structural corruption* into a *clean, detectable rejection*. That alone is the single most important fix for the RWITM hazard. It is OQ11 — resolved, and part of the core rebuild. **Decided — the concrete design and its nine implementation requirements are in §3.11.**
- **Layer 1 — declared, visible modify-intent (the RWITM idea, done softly).** Add an optional declared intent — `get_document(intent: "modify")` — that records a soft, short-TTL **modify reservation** keyed on the file. Critically it does **not** hard-block anyone: it is advisory and informational, because FlashQuery cannot afford a hard lock across think-time. Its value is *visibility* — another caller, and especially an orchestrator or agent loop dispatching work, can see "this file has a pending intended modification, reserved N seconds ago" and choose not to send a second agent at the same file, or to wait. This is RWITM's "declare intent at read" expressed as a signal rather than enforced exclusion. The reservation is a lease — it auto-expires — so a crashed or distracted holder cannot freeze the file. This was **OQ18 — dropped** (§3.14): with the version-fingerprint check (§3.11) covering RWITM correctness, a read-time reservation would add only efficiency, and it is not a requirement for this work.
- **Layer 2 — merge instead of reject (optional; reduces wasted work).** When Layer 0 detects a conflict, rather than a flat rejection, optionally attempt a 3-way merge: base = the version the caller read, theirs = the caller's new content, ours = the current file. Non-overlapping changes both survive; only a genuine same-region conflict rejects. This recovers the main weakness of pure optimistic control — wasted language-model work when the intervening change did not actually collide. It is the highest-effort layer and can be deferred. This is **OQ19 — deferred** (§3.14): the reject-and-retry baseline is decided; the merge is a later enhancement, not part of this work.

**Two structural points that shrink the problem.** First, *granularity shrinks the blast radius.* FlashQuery's surgical tools (`replace_doc_section`, `insert_in_doc`) already re-resolve their target *inside the write lock* against fresh content (§3.8), so a section-targeted edit is never applied to a stale structure. The residual hazard is purely semantic — the caller *chose* its new section text from a stale reading of that section. One way to address that would be **section-level version tokens** (a per-section hash); it was considered and **rejected as overkill** (§3.11, requirement #8). The simpler resolution adopted instead: the whole-file token stays the only token, and a refused write returns the caller's current targeted region (§3.11, requirement #5) — so a caller whose own section was untouched recognises the conflict as unrelated and retries without re-reading, and a caller whose section *did* change sees the new text and re-plans. The server stays simple; the caller does the section-level reasoning.

Second, *prefer surgical writes over whole-file rewrites.* The RWITM blast radius is proportional to a write's footprint: a `write_document` full-body rewrite clobbers everything an intervening writer did; a `replace_doc_section` clobbers only one section. Part of the answer is therefore tool design and caller guidance — steer callers toward the narrow surgical tools and reserve whole-file `write_document` for genuine whole-file replacement. The narrower the write, the less an RWITM collision can corrupt.

This section deepens OQ4 (the stale-reference hazard) and OQ11 (the `expected_version` interface); the further questions it raised — OQ18 and OQ19 — are dispositioned in §3.14 (dropped and deferred respectively).

### 3.11 Decided: the version-fingerprint check

The Layer 0 mechanism from §3.10 has been decided. The design: every `get_document` returns the file's content hash as a fingerprint; any write tool optionally accepts that hash back; on write, the server compares the passed-in hash against the document's current hash; if they match, the file is unchanged since the caller read it and the write proceeds; if they differ, the write is refused and the caller is told the file changed underneath it, so it can re-read and retry.

This is sound, and it is the **ETag / `If-Match` optimistic-concurrency pattern** — the mechanism HTTP, REST APIs, and document databases use for the lost-update problem. A key elegance of the framing: the hash returned on *read* carries no obligation — it is just a fingerprint. Intent to modify is declared **only on the write**, by the caller choosing to pass the hash back as a precondition. There is no separate "read with intent to modify" step; a plain read stays a plain read. This keeps the mechanism opt-in and backward-compatible — existing callers that pass no hash behave exactly as today.

FlashQuery already has the materials. It computes a SHA-256 hash of raw file content on every read (`document-output.ts:420`) and stores it in `fqc_documents.content_hash`. The gap is only that the `get_document` response envelope today carries `identifier, title, path, fq_id, modified, size` — **not the hash**. Surfacing it is the one additive change on the read side. The returned field should be named for its *use*, not its implementation — e.g. `version_token` — so a caller understands it is an opaque value to hand back on a write, not something to interpret or recompute; the write tools accept it as `expected_version` (or `if_match`). (Every file-affecting tool accepts the token — the destructive and structural tools `remove_document` / `move_document` / `archive_document` / `copy_document` included — decided in §3.15, OQ24.) Both the new response field and the precondition parameter should be documented in the `help: true` output of `get_document` and of every write tool, so the mechanism is discoverable rather than folklore.

**Implementation requirements — the details that make it correct rather than merely plausible:**

1. **Compare inside the write lock, against a freshly recomputed hash of the file on disk.** The check is a classic time-of-check-to-time-of-use hazard: if the current hash is read before the write lock is held, another writer can slip in between the check and the write. The sequence must be: acquire the per-file write lock → re-read the file from disk → compute its hash now → compare to the caller's `expected_version` → write or refuse → release. FlashQuery's write tools already re-read inside the lock (§3.8), so the comparison slots into an existing point in the flow.
2. **Compare against disk, not against the database.** `fqc_documents.content_hash` can lag the file (the scanner refreshes it lazily). The check must hash the actual bytes on disk at write time, not trust the stored DB value.
3. **The fingerprint is the whole raw file** — frontmatter plus body. `get_document` should return that whole-file hash even when the caller requested only a section or only frontmatter, because a write affects the whole file. (Section-level hashes, if added later, are *additional, separately labelled* fields — see point 8.)
4. **A successful write returns the new hash.** After a write the file has changed, so its fingerprint changed. The success response should carry the post-write hash, so a caller making a sequence of edits can chain them without a re-read between each.
5. **A refused write returns the new `version_token` and the caller's current targeted region** — decided (OQ20). Not a bare error: the new token always comes back so a retry can succeed without guessing it, and the current content of the *region the caller was operating on* comes back so recovery costs no extra round-trip. With it the caller holds three things — (a) the region as it first read it, (b) the region it intended to write, (c) the region as it now stands — and decides locally: if (a) and (c) are identical the conflict was caused by a change elsewhere in the file and the caller simply retries its write with the new token; if they differ the caller re-reads and re-plans. Three details the spec must pin down: *(i) "targeted region" is per-tool* — the section for `replace_doc_section`, the frontmatter for `apply_tags` and `insert_doc_link`, the whole document for `write_document` (so whole-file callers still get everything), the anchor section or document end for `insert_in_doc`, and the whole document for the destructive/structural tools `remove_document` / `move_document` / `archive_document` / `copy_document` (a conflict on those means "the file changed — the operation was not performed"; see §3.15, OQ24); *(ii) the not-found variant* — if the racing change renamed or removed the caller's target, the response must say so explicitly rather than return a wrong or fuzzily-matched region, and the caller then re-reads the whole document; *(iii) representation consistency* — the region in the refusal must be byte-identical in form to what `get_document` returns for that region, or the caller's (a)-versus-(c) comparison will spuriously fail. The tool already holds the full current file in memory at refusal time (it re-read it under the lock), so this is a response-shaping choice, not extra I/O.
6. **It composes with the locks; it does not replace them.** The per-file write lock prevents two writes from *interleaving* into a corrupt state; the fingerprint check prevents a write built on a *stale read*. They address different failures and are both needed — the check lives *inside* the lock's critical section (point 1).
7. **It is agnostic to who changed the file.** Because the check compares hashes of bytes on disk, it also catches edits made *outside* FlashQuery — by Obsidian, an editor, `git` (the writer the lock can never see, §3.9). A caller that read a file an external tool then edited will be correctly refused. This is a free and valuable property.
8. **The token is a whole-file hash — and that is the final design, not a baseline.** It fingerprints the entire raw file; it is not section-scoped. A consequence is that an edit anywhere in a file invalidates the token for the whole file, so a caller's `replace_doc_section` can be refused even when its own section was untouched (an unrelated edit elsewhere). Section-level tokens — a per-section hash the server would maintain and check — were considered and **rejected as overkill**: they add real server complexity for a problem the conflict-response design already handles. Because a refusal returns the caller's current targeted region (requirement #5), the caller can see its section is unchanged, clear that false conflict itself, and retry. The server stays simple — one hash per file — and the caller does the only fine-grained reasoning that is actually needed. Whole-file token plus region-returning refusal: the two decisions fit together.
9. **The guarantee is opt-in.** Because passing the hash is optional, a caller that omits it can still clobber. That is acceptable and intentional — it mirrors `If-Match` being optional in HTTP, and preserves backward compatibility — but state it plainly: the no-silent-lost-update guarantee holds for callers that pass the fingerprint, not universally. Whether to make it default-on for agent/skill callers was OQ23 — **resolved (§3.15): it stays opt-in.** There is no server-side default-on mode; FlashQuery's own agent and skill callers are expected to pass the token in their own code, which is where the responsibility belongs.

One note on what *not* to use: the response already carries a `modified` timestamp, tempting as a cheaper version token. It should not be used for the check — timestamps have one-second resolution, so two writes within the same second produce the same `modified` value and the second silently passes a check it should fail. The content hash is the reliable fingerprint; `modified` stays informational.

**Token stability — does a scan invalidate the token?** A natural worry, raised in review: the fingerprint hashes the *whole* file including frontmatter, and FlashQuery's scanner reads every file — if a scan rewrote frontmatter, every outstanding token would be invalidated on every scan cycle and the scheme would collapse into constant false conflicts. A codebase check (`services/scanner.ts`) confirms this does **not** happen. A scan of an already-tracked file whose content is unchanged performs **zero writes** — it reads the file, computes the hash, compares it to the stored DB row, and if they match it does nothing. The only scan-triggered vault writes come from `repairFrontmatter()`, which writes *only* files the scan flagged `needs_frontmatter_repair`, and that flag is set in exactly four cases (`scanner.ts:684, 806, 950, 985`): a newly-discovered file with no identity, a duplicate/copied file, or a path-based reconnect — files whose identity frontmatter is genuinely missing or wrong. A steady-state tracked file is never flagged and never rewritten, and `repairFrontmatter` clears the flag after writing so a repaired file is not rewritten on later scans. `fq_updated` is bumped only by real `writeMarkdown` calls, never by a scan-read. So a token stays valid across any number of scans. The one exception is benign: when `repairFrontmatter` writes a new/duplicate/reconnected file it does change that file's frontmatter (it is adding or correcting the `fq_id`), so a token taken just before that one-time repair will correctly invalidate — but the file genuinely changed identity, the outcome is a safe re-read, and it is a single transient event per file, not recurring churn. **This is a load-bearing property and should be made an explicit invariant**: the scanner must never rewrite an unchanged tracked file, and a regression test should assert that two consecutive scans of an untouched vault produce zero file writes. A future change that made scans normalize or re-timestamp frontmatter would silently break the version-token scheme.

With this decided, **OQ11 is resolved.** It also reframes OQ18 (a declared read-with-intent reservation): the fingerprint check delivers the *correctness* — no silent lost update — with nothing declared at read time. A read-time intent signal would now add only *efficiency* — letting an orchestrator avoid dispatching two writers at one file before either does expensive work — so OQ18 drops from "needed" to "optional optimization."

### 3.12 Decided: the locking architecture

OQ2, OQ9, OQ10, OQ17, and OQ21 are resolved here — they were always facets of one decision. This is the locking architecture for the vault-write rework; §3.7 is the staged plan to reach it.

**Granularity — per file.** The lock unit is the individual file. Writes to two different files never block each other; only writes to the *same* file serialize. This replaces today's single global `'documents'` lock, under which every document write in the vault contends on one lock (§3.2–§3.3).

**The lock key — the canonical absolute filesystem path.** Every file is identified by its absolute path, resolved with `realpath` so that symlinks and `.`/`..` segments cannot fragment one file into several keys. Not the vault-relative path — that is ambiguous when two vaults overlap on disk, so two processes would take different locks for the same file (OQ21). Not keyed on `instance_id` — the same physical file must lock identically regardless of which instance or vault reaches it.

**Two tiers.** A writer acquires Tier 1, then Tier 2; it releases in reverse order.

- *Tier 1 — in-process.* A striped registry of `async-mutex` locks, keyed by the canonical absolute path, in the FlashQuery process's memory. It serializes concurrency *within* one FlashQuery process — many MCP sessions interleaving on one event loop. Cost: microseconds, no I/O. It is a performance fast path, not a correctness layer: removing it would leave the design correct, only slower.
- *Tier 2 — global, cross-process.* Native Postgres advisory locks — session-scoped `pg_advisory_lock(key)` / `pg_advisory_unlock(key)`, taken on a connection checked out from the raw `pg` pool that already exists (`utils/pg-client.ts`). The key is the canonical absolute path hashed to a bigint. It serializes concurrency *between* separate FlashQuery processes that share a vault — the correctness layer. Because Tier 1 collapses all same-process contention, Tier 2 is reached at most once per process.

**Folder operations — shared/exclusive directory locks (OQ13).** Per-file locks protect individual files; folder-structure operations (renaming, moving, or deleting a folder) act at a coarser granularity and need their own coordination, or a folder rename could move a file out from under an in-flight write. The protocol: a file write holds a **shared** lock on every ancestor folder, from the file's parent up to the vault root; a folder structural operation takes an **exclusive** lock on the folder it changes. Shared locks coexist, so concurrent file writes — and reads, which take no lock at all — are never blocked by one another; an exclusive folder lock conflicts with the shared locks beneath it, so a folder operation automatically waits for in-flight descendant writes and blocks new ones until it completes. The relationship is symmetric: while a write is in flight, structural change to any ancestor folder is blocked; while a folder is being restructured, descendant writes are blocked. Folder *creation* is additive — it invalidates no existing path — and does not take the exclusive lock. This is multiple-granularity locking reduced to plain shared/exclusive; full IS/IX/SIX intention locks were considered and rejected as overkill for an operation this rare. Directory locks live in the **global tier** only — Postgres advisory locks have a native shared mode (`pg_advisory_lock_shared`) — keyed on the folder's **canonical absolute path**, exactly as file locks are: this makes the protocol correct across processes, across instances, and across overlapping vaults (the same physical folder reached through two vaults resolves to one key), and self-scoping, so non-overlapping vaults never falsely contend. The in-process tier stays file-only. `manage_directory` (`files.ts`) — today the lone `directory:`-keyed `fqc_write_locks` user (§3.2) — is the tool that takes the exclusive folder lock; retiring `fqc_write_locks` migrates it onto these advisory directory locks.

**`fqc_write_locks` is retired.** The hand-rolled lock table reimplemented advisory locks as application data, and did it worse: the mis-keyed primary key (OQ9) gave no real cross-instance exclusion; client-side exponential-backoff polling stood in for native blocking; and a manual `expires_at` TTL plus the `flashquery unlock` CLI substituted for automatic release. Native `pg_advisory_lock` provides correct exclusion, native blocking, a bounded wait (§3.7 step 8, OQ29), and automatic release when the holding connection ends — crash recovery for free, no TTL to tune. The OQ9 primary-key defect is not patched; it disappears with the table. (An earlier draft objected that advisory locks "do not fit FlashQuery's data access" because the Supabase client is REST-based; that objection was wrong — FlashQuery already carries a raw `pg` connection pool, so advisory locks need no new infrastructure.) Retiring the table includes removing it: on FlashQuery startup, `fqc_write_locks` is dropped if it exists, with a single `debug`-level log line recording the removal — no per-operation checking, and, with effectively no current users, no backward-compatibility concern.

**What this guarantees:**

- *Mutual exclusion per file* — across async contexts, across processes, across instances — because the key is the physical file and Tier 2 is visible to every process sharing the database.
- *Bounded-wait fairness for writers* — every contended writer either acquires the lock or fails cleanly within the OQ22 timeout, so no writer is starved indefinitely; strict global FIFO is **not** promised — Tier 2 advisory locks carry no documented waiter-ordering guarantee (§3.7 step 8, OQ29, §3.16).
- *Reads never block* — the lock is write-only; atomic-rename (§3.5) already gives every reader a complete-old-or-complete-new view.
- *Crash safety* — an advisory lock releases automatically when its connection drops, so a crashed writer cannot strand a lock.

**Caveats carried into the spec:**

- Tier 2 holds a checked-out `pg` connection for the duration of each cross-process-contended file write. Because Tier 1 makes Tier 2 reachable at most once per process, the count of simultaneously-held lock connections is bounded by the (small) number of processes — pool exhaustion is not a practical risk. Tier 2 does require a session-capable connection (a direct connection or a session-mode pooler, not a transaction-mode PgBouncer).
- The advisory-lock key is a bigint hash of the path; two different paths can hash-collide, which would make two unrelated files occasionally wait on each other — harmless false sharing, the same trade-off as lock striping, never a missed exclusion.
- Hard links and bind mounts can still alias one physical file under two absolute paths that `realpath` will not unify; that aliasing across vaults is a documented unsupported configuration (OQ21).

The version-fingerprint check (§3.11) runs inside the Tier-1-plus-Tier-2 critical section; cooperative chunking for bulk passes (§3.7 step 5) builds on this lock.

### 3.13 Decided: the multi-file batch contract

Several write tools act on more than one file in a single call — `insert_doc_link` (a pointer into N source documents), `apply_tags`, `archive_document`, `remove_document`, and any `call_macro` that touches multiple files. The batch form exists purely for **performance** — it spares the caller N separate MCP round-trips — not to provide atomic write grouping. The contract reflects that.

**A batch is N independent single-file operations, run faster — not a transaction.** Each file is written exactly as a single-file call would write it: under its own per-file lock and ancestor shared directory locks (§3.12), with its own optional version-fingerprint check (§3.11). Items are independent — one item's outcome does not affect another's. The batch is **not** atomic and gives **no ordering guarantee**: the engine may process items in input order, skip past an item briefly waiting on a lock and return to it, or process them in parallel — all valid, because correctness never depends on batch order or all-or-nothing semantics. (This is the saga pattern — per-item work, no distributed transaction.) Folder coordination needs nothing special: each item takes its ancestor shared directory locks per §3.12, exactly as an individual write would.

**Why best-effort, not atomic.** FlashQuery's batch operations are additive and idempotent — a half-applied batch is *unfinished*, not *corrupt*; every file in it stays individually valid. True multi-file atomicity is a database transaction, which a filesystem does not provide; emulating it (save-and-restore rollback) is fragile — a crash mid-rollback is worse than the partial state. And the contract must be uniform across every batch tool — hand-rolling rollback five-plus times is untenable. Best-effort is also already FlashQuery's de-facto behaviour: `remove_document`'s contract today says "batch responses preserve input order and report per-document errors." This formalises and unifies an existing pattern rather than inventing one.

**The response envelope.** A batch returns an **ordered per-item result array** — one entry per input file, in input order — and each entry is exactly the shape a single-file call of that tool would return:

- *succeeded* — carries the file's new `version_token`, so a caller chaining edits need not re-read.
- *conflicted* — the item carried a `version_token` and it no longer matched (the RWITM case: another process changed the file). The entry carries the new `version_token` and the caller's current targeted region — the identical recovery payload defined for a single-file conflict in §3.11, requirement #5.
- *failed* — a non-conflict error (not found, invalid path, etc.) with its reason.

A batch with no `version_token`s behaves exactly as today — every item is written best-effort, with brief invisible pauses if an item's lock is momentarily held — and the response is the per-item array already in use. `version_token`s are the opt-in: when present, conflicted items appear in the array with their recovery payload, and the caller retries **just those items** (re-read, re-plan, re-issue) — idempotency makes a targeted retry safe. Two kinds of "pending" must not be confused: an item *waiting for a file's lock* is brief, bounded, and invisible to the caller (absorbed as latency); an item *failing its version check* is an immediate refusal that appears in the array. Only conflicts and hard failures reach the caller. The spec must pin down, per tool, what a conflicted entry's "targeted region" is — the frontmatter for `insert_doc_link` / `apply_tags` / `archive_document`; for `remove_document` a conflict means "the file changed — it was not removed," so the entry carries the new token and the current content — the same per-tool definition §3.11 requirement #5 already calls for.

**Atomic batches — a reserved opt-in.** No current built-in tool needs all-or-nothing semantics. If a future operation genuinely corrupts on a partial result — the plausible home is a `call_macro` whose steps only make sense together — that operation may opt into atomic execution: acquire every per-file lock up front in sorted canonical-path order (deadlock-free — the "lock at the outset" rule), with a cap on how many locks one operation may hold. This is a per-operation opt-in, not the default, and is not built until a concrete need exists.

### 3.14 Out of scope and deferred work

Five questions raised during the research are deliberately **not** requirements for this work. They are recorded here — with their dispositions and reasoning — so the requirements spec excludes them cleanly and nothing is lost.

**OQ14 — cross-document pointer identity — out of scope.** FlashQuery's cross-document pointers are written today by *title* (`insert_doc_link` builds a `[[targetTitle]]` wikilink) or by *path* (`relates_to` / `depends_on` frontmatter). Both are *mutable* identifiers: a concurrent rename or move of the target silently invalidates every pointer to it, and no lock can prevent it — the captured title or path was already wrong the moment it was captured. The structural fix is to reference the stable `fq_id` UUID FlashQuery already assigns each document and resolve to a display title/path at read time, which makes a rename a non-event for the link graph. This is a genuine improvement — but it is a *separate feature*, the link/reference model, not part of the locking and write-coherency rework. **Disposition: out of scope — not a requirement for this work.** Recommended: track it as its own roadmap/research item so it is not lost.

**OQ18 — declared read-with-intent reservation — dropped.** It was considered whether `get_document` should offer an `intent: "modify"` mode that records a soft, short-TTL "modify reservation," so an orchestrator could see a pending intended modification and avoid dispatching a second writer at the same file. **Disposition: not a requirement — dropped.** RWITM (the read-to-write gap) is handled in full by the version-fingerprint check (§3.11), which needs nothing declared at read time: a stale write is caught and refused at write time. A read-time reservation would add only *efficiency* — avoiding wasted work *before* a collision rather than detecting it after — never correctness. With correctness already covered by a mechanism that requires no declared intent, it is dropped, not merely deferred. (This was "Layer 1" in §3.10's layered framing.)

**OQ19 — merge-on-conflict — deferred enhancement.** On a version-conflict the decided behaviour is **reject-and-retry**: refuse the write and return the recovery payload (§3.11, requirement #5) so the caller can re-read and re-issue. That is decided and in scope. A *3-way / section-aware merge* — completing a caller's write automatically when the intervening change did not overlap it, instead of refusing — was considered as a way to avoid wasting a language model's generated work on a non-conflicting collision. It is the highest-effort option, it is not needed for correctness, and it is **deferred**: revisitable once the core mechanism (§3.11–§3.13) is built and in use. The reject-and-retry baseline stands on its own without it. (This was "Layer 2" in §3.10's layered framing.)

**OQ36 — macro-engine auto-threading of `version_token` — deferred enhancement.** When a macro reads file X (`$x = get_document(path: "X.md")`) and a later step writes the same file without an explicit `expected_version`, the engine would substitute `$x.version_token` automatically — delivering multi-step RWITM safety without burdening the macro author. The right long-term direction, but it requires an identifier-equivalence design (when are `path`, `fq_id`, and a filename-only identifier "the same file"?), and it is a semantic change to existing macros (a macro that succeeded silently today might start refusing writes once auto-threading lands) that deserves its own announcement and migration path. **Disposition: deferred** — not bundled into the locking rework. See §3.17 for the full discussion.

**OQ37 — `call_macro` atomic-execution opt-in — deferred enhancement.** An optional `atomic: true` flag on `call_macro` that takes per-file locks on every file the macro will touch in sorted canonical order before any step runs, holds them for the macro's lifetime, and releases at the end — the macro-shaped extension of OQ19/§3.13's atomic-batch opt-in. Solves multi-step consistency for macros that opt in; constrained by the requirement that the file set be knowable up front (which rules out wildcard macros). **Disposition: deferred** — built when a concrete macro genuinely needs it, ideally co-designed with the atomic-batch opt-in. See §3.17 for the full discussion.

### 3.15 Decided: product-behavior semantics

The 2026-05-25 verification pass surfaced four caller-visible behaviors the architecture had left open — product behavior a requirements spec must state outright, not implementation detail. All four were decided the same day. None changes the locking architecture (§3.12), the version-fingerprint check (§3.11), or the batch contract (§3.13); each pins a behavior down.

**OQ22 — a contended write times out; it does not block indefinitely.** Tier 2 (`pg_advisory_lock`) is acquired with a *bounded* wait, not the default indefinite block — mechanically Postgres `lock_timeout`, or `pg_try_advisory_lock` wrapped in a bounded retry loop. On timeout the write fails and the failure is reported in the return object with a clear, catchable message (a lock-timeout / "resource busy" error an agent loop can detect and act on). The timeout is **10 seconds by default**, configurable in `flashquery.yml` under the locking section with explanatory commentary; when the setting is left commented out it defaults to 10 s. Rationale: an LLM-driven caller cannot notice it is stuck behind a slow writer, so an unbounded wait is the wrong default, and 10 s keeps the order of magnitude of the retired `fqc_write_locks` timeout.

**OQ23 — the version-fingerprint check is opt-in.** A caller that passes `version_token` is, by that act, opting into the check; a caller that omits it writes unconditionally, exactly as today. There is no separate default-on mode, and so no `get_document`→write token-threading machinery to build — the surface stays minimal and fully backward-compatible. The protection the earlier "default-on for agent/skill callers" idea aimed at is still achievable, but by having FlashQuery's own agent and skill callers pass the token in *their* code, not by a server-side default — responsibility sits with the caller, which is the correct place for it. §3.11 requirement #9 already states the consequence plainly (the no-silent-lost-update guarantee holds for callers that pass the fingerprint, not universally) and stands as the final word.

**OQ24 — every file-affecting tool accepts an optional `version_token`, destructive tools included.** The token is accepted not only by the content-editing tools (`write_document`, `insert_in_doc`, `replace_doc_section`, `apply_tags`, `insert_doc_link`) but also by the destructive and structural tools `remove_document`, `move_document`, `archive_document`, and `copy_document`. The token always refers to *the file the caller read and is acting on*: for `remove_document` / `move_document` / `archive_document` it is the file being removed, moved, or archived; for `copy_document` it is the source file. Rationale: an action taken from a stale view is most costly precisely when it is destructive — the decision to delete a file is exactly the decision that should change if something updated that file a moment ago. The token stays optional everywhere, so a caller that does not care simply omits it. This resolves the §3.11-vs-§3.13 inconsistency the verification pass found: §3.11 requirement #5's per-tool "targeted region" definition now carries a row for each destructive tool — a conflict there means "the file changed — the operation was not performed," and the refusal returns the new token plus the current file content, the same recovery payload §3.13 already specifies for a `remove_document` conflict.

**OQ25 — records, memory, and plugins are outside the file-locking scope; a completed per-call-site audit records what retiring `fqc_write_locks` requires.** These three subsystems hold the coarse `'records'` / `'memory'` / `'plugins'` locks today, but they write database rows, not vault files — per-file locks do not apply to them. A full audit of every locked call site (2026-05-25) found:

- *Memory (`write_memory`).* The `'memory'` lock is **redundant**. Create mode is a single atomic `INSERT` into `fqc_memory`; update mode calls the `fqc_memory_create_version` RPC, which is transactional and carries its own concurrency control — `SELECT … FOR UPDATE` plus a uniqueness constraint on the latest-version row, with the tool already handling the `23505` / `P0002` races explicitly. The RPC is the real guard; retiring the coarse lock removes nothing memory depends on.
- *Records (`write_record`, `archive_record`, `search_records`).* The record writes themselves are single atomic SQL statements (`INSERT`, or `UPDATE … WHERE id = …`); Postgres serializes concurrent same-row writes natively, with the same last-writer-wins outcome the coarse lock produced. What the `'records'` lock actually serializes is the **reconciliation preamble** (`reconcilePluginDocuments` + `executeReconciliationActions`) shared by all three handlers — a multi-step detect-plan-act that *writes plugin documents*. (`search_records`, a read tool, holds the *write* lock solely for this preamble.) Reconciliation's document writes will fall under per-file locks once the document rework lands; its detect-plan-act decision will not. Requirement for the rework: before dropping `fqc_write_locks`, confirm reconciliation is idempotent under concurrent runs, or give it its own guard.
- *Plugins (`unregister_plugin`).* This is the only `'plugins'` lock user — `register_plugin` does not take it, so the lock never protected register-against-unregister. `unregister_plugin` is a genuine non-atomic multi-statement sequence: count records → conflict-check → clear document ownership → delete plugin-scoped memories → delete pending reviews → delete the registry row. Plugin (un)registration is a rare admin action and the realistic collision — two concurrent unregisters of one plugin — is largely idempotent (`DELETE`s of already-gone rows). Requirement for the rework: give this delete sequence an explicit guard when the table is retired.

Where a guard is still needed — the reconciliation preamble and the `unregister_plugin` sequence — the decided replacement is a **per-plugin-scoped Postgres advisory lock** (the same native `pg_advisory_lock` mechanism as the file tier in §3.12, keyed on `plugin_id` rather than a file path), or a wrapping transaction for the `unregister_plugin` deletes. Memory needs nothing. No coarse lock survives in `fqc_write_locks`'s place. §3.7 step 1 is corrected accordingly — it no longer implies per-file locks for records / memory / plugins.

### 3.16 Decided: codebase-review refinements (OQ26–OQ34)

A 2026-05-25 review against the current `flashquery/` source produced nine follow-ups (OQ26–OQ34); every code observation was verified accurate. Six are genuine pre-spec questions, decided here with Matt; OQ26 is a citation cleanup (done); OQ33 and OQ34 are devspec-scope requirements with predetermined answers, recorded so the spec absorbs them. These decisions refine — they do not reopen — §3.11–§3.13.

**OQ26 — citation sweep (done).** The monolithic `src/mcp/tools/documents.ts` has been split into `documents/write.ts`, `get.ts`, `remove.ts`, `archive.ts`, `copy.ts`, `move.ts` (plus `deps.ts` / `helpers.ts`); `documents.ts` is now a registration barrel. The stale `documents.ts:<line>` citations in §3 and §7 were re-pointed to the split files (`write_document` is `documents/write.ts`, with `acquireLock` at line 66 and `releaseLock` at 312). `compound.ts`, `records.ts`, `memory.ts`, `plugins.ts`, and `files.ts` were not split and keep their citations. The dev agent should confirm exact line numbers in the split files against the code.

**OQ27 — a read's `version_token` is always the SHA-256 of the file's on-disk bytes at the moment the response is built; file, DB row, and token are kept mutually consistent.** `get_document` does not unconditionally scan: it reads the file and hashes it, and *only when* that hash differs from the stored `fqc_documents.content_hash` (or no row exists) does it run `targetedScan`, which may write a frontmatter repair. The verified defect: today the hash computed *before* any repair (`document-output.ts:420`) is the value both written to the `fqc_documents` row (`:465`) and — once surfaced — returned as the token; after a repair, file, DB row, and token all disagree with the bytes now on disk. The decision: whenever a repair writes the file, the hash is recomputed from the *exact bytes written*, and that single post-repair value is used everywhere a post-state hash is needed — the returned `version_token` and the `content_hash` row update — so all three agree. Implement it at the write helper (`writeMarkdownFile` / `targetedScan` returns the hash of what it actually wrote), not as a re-read bolted onto every reader. To the question raised in review: yes — the hash is (re)computed at `get_document` time and the DB row is updated to match — but this is *not* a new unconditional `targetedScan`; the scan and the row update already run only on drift, the common no-drift case still does zero writes, and the change only ensures that *when* a repair occurs the reported hash reflects post-repair disk state. A test asserts a `get_document` that triggers a repair returns a token a subsequent no-op write accepts. Refines §3.11 (the token-stability invariant).

**OQ28 — the Tier 2 lock tier requires a session-capable Postgres connection, validated at startup.** Session-scoped `pg_advisory_lock` is correct only on a connection whose backend session is stable — a direct connection or a session-mode pooler — never a transaction-mode pooler (Supabase's Supavisor in transaction mode hands each transaction a different backend, stranding the lock on an unrelated client). The decision: require a session-capable connection for the raw `pg` pool that hosts Tier 2 (`utils/pg-client.ts`), and validate it at startup with a self-test — take a throwaway advisory lock, confirm it is observable, release it — failing loudly with a clear message otherwise. Transaction-scoped `pg_advisory_xact_lock` was rejected: it would hold a DB transaction open across the file write, the long-held-resource hazard the design exists to avoid. **Downstream changes the devspec and test plan must codify** (raised in review): the README / setup docs state the session-capable-connection requirement; `.env` / `.env.example` document that `DATABASE_URL` (the raw `pg` pool's connection) must be the direct or session-mode endpoint, distinct from any transaction-pooled URL; `flashquery.yml` carries commentary in the locking section; the startup self-test and its error message are implemented; the Supabase onboarding / deployment guide is updated; and the test plan includes a case asserting a transaction-mode pooler connection is detected and rejected. Refines the §3.12 caveat, promoting it from a note to a firm, tested requirement.

**OQ29 — FlashQuery promises a bounded wait, not strict FIFO.** §3.7 step 8 and §3.12 previously claimed `pg_advisory_lock` delivers FIFO writer ordering; PostgreSQL does not document FIFO as a guarantee for advisory-lock waiters (the lock manager may reorder them, e.g. to break deadlocks). The claim is corrected: the guarantee is a **bounded wait** — every contended writer either acquires the lock or fails cleanly within the OQ22 timeout (10 s default), so no writer is starved indefinitely; Tier 1's `async-mutex` remains genuinely FIFO *within* a process, but no strict global FIFO is promised. No custom wait-queue is built — the timeout already delivers the liveness an agent caller needs. §3.7 step 8 and §3.12 are corrected in place.

**OQ30 — batch tools take a mixed identifier array to carry per-item `version_token`s.** Every batch-capable parameter that today accepts `string | string[]` — `remove_document` and `archive_document` (`identifiers`), `insert_doc_link` source identifiers, `apply_tags` targets, and any future batch tool — is widened to `Array<string | { identifier, version_token }>`: a bare string is an untokened item (behaves exactly as today), an object form carries the token. Tokened and untokened items may be mixed in one call; an untokened item simply skips the version check. This is fully backward-compatible and mirrors the per-item *output* envelope §3.13 already defines. Parallel positional arrays and a separate identifier→token map were both rejected (off-by-one fragility; broken when an identifier repeats or is itself path-like). Refines §3.13 — this is the batch contract's *input* shape, the complement to its output envelope.

**OQ31 — operations that create or move into a destination lock the destination path, not only the source.** A per-file lock keyed on the source path leaves a check-then-act race on destinations: `copy_document`, `move_document`, and create-mode `write_document` each test destination existence and then write, so two callers can both pass the check and race into the same path. The decision: a tool that writes a destination takes the per-file lock on that **destination** canonical path for the operation's duration; `move_document` locks **both** source and destination, acquired in sorted canonical-path order (the §3.13 deadlock-free rule); ancestor shared directory locks (§3.12) compose on top. The devspec carries a per-tool table — which path(s) each tool locks, and in what order. Refines §3.12 — the lock model now explicitly covers destination and not-yet-existing paths. Depends on OQ32 for the destination-key form.

**OQ32 — the canonical lock key: `realpath` for existing paths, `realpath(parent)+basename` for destinations, case-folded, namespaced.** §3.12's "canonical absolute path" is made precise. For an existing file or directory the key is `realpath(path)`. For a not-yet-existing destination — a create / copy / move target — it is `realpath(parentDir) + '/' + basename`, because `realpath` cannot resolve a path that does not exist yet. On a case-insensitive filesystem (macOS APFS by default, and Windows) the key is case-folded; otherwise `Notes/Plan.md` and `notes/plan.md` — the same file on disk — would take different advisory-lock keys and silently lose a mutual exclusion. A resource namespace (`file:` / `dir:`) is prefixed before the key is hashed to the advisory-lock bigint. A test asserts two case-variant paths to one file take the same lock. Refines §3.12 (the lock key).

**OQ33 — every vault write routes through one lock-aware, durable atomic write primitive (devspec scope, not a deliberation).** The review confirmed at least three independent write paths — `vaultManager.writeMarkdown`, the `writeMarkdownFile` helper inside `targetedScan`, and `atomicWriteFrontmatter` (which still swallows write errors at `debug` level) — plus the `move_document` EXDEV fallback (OQ34). There is no decision to make: a write path that bypasses the lock means the lock is not enforced. The requirement: the devspec includes an exhaustive write-path inventory — every `writeFile` / `rename` touching the vault — as an explicit work item, and every one routes through the single lock-aware, unique-temp-name, durable primitive. This composes with §3.5 (durability, and the silent-failure fix: `atomicWriteFrontmatter` must surface errors, already decided there) and §3.7 step 6 (unify the mechanisms).

**OQ34 — the cross-device (EXDEV) copy/move fallback uses the same atomic+durable discipline (devspec scope, not a deliberation).** `move_document`'s EXDEV fallback today does `readFile` → direct `writeFile(dest)` → `stat` → `unlink(source)`, which can leave a partial destination after a crash. The correct implementation is fully determined: write a unique-named temp file in the *destination* directory, fsync it, rename it into place, then remove the source only after the destination is committed — the §3.5 atomic+durable write applied to the cross-device path. It is folded into the §3.5 / §3.7 step 4 durability work as one more write path that must use the standard primitive (tying it to OQ33); it is not a separate product decision.

### 3.17 Decided: macro-level concurrency model (OQ35–OQ37)

The vault-write rework is uniform at the per-tool layer: every file-affecting tool acquires a per-file lock and (optionally) honors a `version_token` precondition (§3.11–§3.12). `call_macro` (`src/mcp/tools/macro.ts`) is one of those callers — it executes a user-provided sequence of MCP tool calls, can use wildcards and dynamic discovery, and can run for an extended period — and a natural question is whether that uniform pattern is enough to keep two macros running in parallel from stepping on each other, or whether the macro layer needs its own coordination on top. The conversation that produced this section (2026-05-25) walked through that question; this section records the analysis and the dispositions so future review can revisit it cleanly.

**What `call_macro` does today.** A code check confirms `call_macro` takes **no** write lock at the macro level — `src/mcp/tools/macro.ts` does not call `acquireLock` or any equivalent. The handler parses the macro source, evaluates it through `evaluateProgram` (`src/macro/evaluator.ts`), and dispatches each tool call through the MCP broker. Once Phase 1 of the rework wires `withDocumentLock` into every documents/* and compound.ts call site, every `write_document` / `replace_doc_section` / `apply_tags` / etc. inside a macro automatically takes the per-file lock — there is no macro-shaped hole in the lock layer, because the engine never bypasses the tool dispatch.

**What the uniform pattern protects against — no extra design needed.**

- *Two macros writing to the same file.* The per-file lock serializes the writes; the second macro's tool re-reads inside the lock (INV-10), so its change builds on the first macro's bytes — no torn file, no silent overwrite of the first macro's structural changes at the file level.
- *A macro writing while a concurrent `manage_directory` rename runs against an ancestor folder.* The file write's shared ancestor folder lock (§3.12) blocks the rename until the write completes (or the rename times out per OQ22).
- *A macro racing the scanner.* Same per-file lock — the scanner adopts the helper at §3.7 step 5/6, so scans and macro-driven writes coordinate on the same key.

**What the uniform pattern does *not* guarantee.**

1. **Multi-step macro consistency without explicit token threading.** A macro that does `get_document A` → think → `write_document A` is the classic RWITM case. The version-fingerprint check (§3.11) is the right tool — but it is **opt-in** (§3.15 OQ23), so the macro author has to explicitly pass `expected_version: $a.version_token` on the matching write. The macro engine does not auto-thread tokens through matching read→write pairs. Two macros running in parallel on the same file, neither threading tokens, both write — last writer wins silently.
2. **No macro-level all-or-nothing across multiple files.** A macro that updates files A, B, and C as a logical unit takes and releases each per-file lock per step. Another macro can slip in between the A and B writes. This matches §3.13's best-effort batch contract and §3.14's deferred atomic-batch opt-in — extending the opt-in to macros would be the natural mechanism if such an extension is needed.
3. **Wildcard / dynamic-target discovery races.** A macro that does `list_vault(pattern: "Notes/**")` at step 1 and then writes per result at step N can find that a target was renamed or removed between discovery and use. Per-file locks and the version-token check protect against *content* drift; file *move/delete* hits the pointer-stability problem (§3.14, OQ14 — out of scope for this work). The realistic outcome today is a clean `not_found` or `version_mismatch` error at the write step, not corruption — acceptable, but worth being explicit about so a macro author knows the boundary.

**Options considered.**

- **Option A — Treat the macro path as another caller. Best-effort, opt-in tokens.** Each tool call in a macro uses the per-file lock and (optionally) the version-token check, same as a direct MCP call. No macro-level lock, no auto-threading, no atomic-execution opt-in. Macro authors who need stronger multi-step safety thread `version_token`s explicitly. *Pros:* uniform; nothing surprising for the dev agent; ships within the existing v1 work; the macro engine stays semantically unchanged. *Cons:* macro authors must reason about RWITM themselves; multi-step consistency is on them.
- **Option B — Macro-engine auto-threading of `version_token`.** When a macro reads file X (`$x = get_document(path: "X.md")`) and a later step writes the same file (`write_document(path: "X.md", …)`) without an explicit `expected_version`, the engine substitutes `$x.version_token` automatically. *Pros:* delivers safety without macro-author burden; arguably the correct long-term direction. *Cons:* needs a careful **identifier-equivalence** model — when are `path: "A.md"`, `fq_id: "abc"`, and a filename-only identifier "the same file" for the purposes of token threading? — that is a non-trivial design conversation in its own right. Also a semantic change: a macro that succeeded silently today might start refusing writes once auto-threading lands, which deserves its own announcement and migration path rather than being bundled into a locking rework.
- **Option C — `call_macro` atomic-execution opt-in.** An optional `atomic: true` flag that takes per-file locks on every file the macro will touch in sorted canonical order before any step runs, holds them for the macro's lifetime, and releases at the end. *Pros:* solves multi-step consistency completely for macros that opt in. *Cons:* macros that use wildcards or dynamic discovery cannot opt in (the file set isn't knowable up front); holding many locks for a long macro is the contention pattern this rebuild is explicitly avoiding for normal writes; power tool with sharp edges; same constraint and same shape as §3.14's deferred atomic-batch opt-in, so the two should ship together when they ship.
- **Option D — Soft warning on stale-read patterns.** The engine detects "read-then-untokened-write of the same path" and emits a log / warning when it sees it. *Pros:* cheap. *Cons:* half-measure; doesn't actually fix the gap; produces noise on perfectly safe macros (read-then-overwrite is a legitimate use case).

**Decision: Option A.** Macro-level concurrency is treated as another instance of the uniform per-file pattern. The rebuild does **not** introduce any macro-spanning lock and does **not** auto-thread `version_token`. Two reasons for not pulling B or C into v1:

1. The macro-engine auto-threading (B) needs the identifier-equivalence design conversation we have not had, and it would be a semantic change to existing macros that deserves its own initiative rather than bundling into a locking rework.
2. The atomic-execution opt-in (C) is the macro-shaped version of §3.14's deferred atomic-batch opt-in; it is the right mechanism if a concrete macro genuinely needs it, but no current built-in macro needs it. It will land naturally when the atomic-batch opt-in lands, and the two should be designed together.

This decision keeps the rebuild scope coherent. The uniform per-step pattern already gives macros strong individual-write safety; macro authors who need multi-step RWITM safety have the explicit `version_token` mechanism available today; and the two deferred extensions are recorded so a future product owner can pull them forward when motivated by a concrete use case rather than by speculative concern.

**Dispositions.**

- **OQ35 — Does `call_macro` hold any macro-level lock spanning the entire macro?** **Resolved.** No. Each tool call inside a macro takes its own per-file lock via `withDocumentLock`; `call_macro` itself acquires no macro-spanning lock. The uniform pattern from §3.11–§3.13 applies. Requirements REQ-025 captures this so the dev agent does not infer otherwise, and a static-check test guards against any future macro-spanning lock being introduced inadvertently.
- **OQ36 — Should the macro engine auto-thread `version_token` through matching read→write pairs?** **Deferred** (§3.14). Correct long-term direction; needs the identifier-equivalence design and is a semantic change to existing macros. Not built in this rework.
- **OQ37 — Should `call_macro` support an atomic-execution opt-in (all-locks-upfront)?** **Deferred** (§3.14). The macro-shaped extension of §3.14's atomic-batch opt-in; same constraint (the file set must be knowable up front, which rules out wildcard macros); built when a concrete macro need surfaces, ideally co-designed with the atomic-batch opt-in.

**What this means for the dev agent.**

- No new locking code in `src/mcp/tools/macro.ts`. A build-time/static check asserts that `macro.ts` does not import lock primitives directly. Each macro-dispatched tool call goes through its own handler, which already uses `withDocumentLock` after Phase 1.
- Macro authors who need RWITM safety across multi-step flows pass `expected_version` explicitly in their macro source — the version-fingerprint check is the recommended pattern.
- The `help: true` output for `call_macro` should mention this: a short paragraph stating "every step takes its own per-file lock; macros are not atomic — the engine does not span any lock across steps; for multi-step safety across read→write pairs, pass `version_token` explicitly; see §3.14 deferred items for the planned future extensions."

**Implementation notes for OQ37 (the deferred atomic-execution opt-in).**

The deferred atomic-execution opt-in does not require any rework of the decided architecture — the existing `withDocumentLocks(paths, fn)` helper (Requirements REQ-009) already provides exactly the right shape, and the bounded-wait timeout (REQ-006) already gives the up-front-acquire path a clean failure mode. The walkthrough below is captured here so the future implementer does not have to re-derive it from first principles.

*Two forms would be supported, and the inline form is the more useful of the two.*

- **Parameter form on `call_macro`.** A `call_macro(source: …, atomic: true, …)` flag. The engine walks the macro AST at start, extracts every path argument from file-affecting tool calls (literal strings and identifiers bound in scope), canonicalizes and sorts them per REQ-003 and INV-09, calls `withDocumentLocks([...sortedPaths], () => evaluateProgram(…))`. Wildcard outer macros (where the file set is not statically determinable at macro entry) reject with a clear error — the same constraint §3.13's reserved atomic-batch opt-in carries.
- **Inline `atomic { … }` block in macro source.** Strictly more useful — it composes with wildcard discovery. Path extraction happens at *block entry*, with the surrounding scope (including loop variables) bound. The natural pattern is:
  ```
  $files = list_vault(pattern: "Notes/**")
  for $f in $files {
    atomic {                                  // paths extracted HERE, with $f bound
      $cur = get_document($f)
      write_document($f, transform($cur))
    }                                         // locks released; next iteration acquires fresh
  }
  ```
  Each iteration locks only `$f`, runs the read-write sequence under that lock, releases. The parameter form would have to reject this outer wildcard; the inline form handles it natively.

*Three pieces of new implementation work.*

1. **Tier 1 re-entrancy.** The one architectural surprise worth flagging. Tier 2 (`pg_advisory_lock`) is re-entrant by default — the same session + same key just counts up, and `pg_advisory_unlock` matching times is correct. Tier 1 (`async-mutex`) is *not* re-entrant — a context taking the same mutex twice deadlocks. So when an atomic block pre-acquires `X.md` and a nested `write_document("X.md")` inside the block calls `withDocumentLock("X.md")`, we self-deadlock unless the helper detects "already held in this async context." The node-idiomatic fix: an `AsyncLocalStorage`-backed set of held lock keys checked at helper entry — if the key is in the set, run `fn` directly (no acquire/release); otherwise acquire, add to the set, run, remove, release. Tier 2 needs no change. Nesting (`atomic { … atomic { … } }` and an inline block inside `call_macro(atomic: true, …)`) becomes a no-op at the inner block by construction — the correct behavior.

2. **Path extraction from the macro AST.** Walk the AST node for the block (inline form) or the whole macro (parameter form) and collect path arguments from every file-affecting tool call. Literal strings are static; identifiers (loop variables, scope-bound values) are resolved at block-entry time, not parse time. A path that comes from a function call *inside* the block — e.g., `write_document(path: pick_random())` — cannot be pre-acquired; the block fails with "atomic mode requires a determinable path set," matching the constraint that already governs §3.13's atomic-batch opt-in.

3. **Macro language syntax** for the inline form: an `atomic { … }` block the parser recognizes and the evaluator dispatches through `withDocumentLocks`. The parameter form needs no language change — just a Zod schema field on `call_macro`'s input.

*Trade-offs to capture in the help text and language docs when the feature ships.*

- The REQ-006 bounded-wait timeout still applies to the up-front `withDocumentLocks` acquire. If any of the N files is contended past the timeout, the whole atomic block fails with a `lock_timeout` envelope — no work runs, no locks held. Clean rollback by construction; macro authors plan for the failure path.
- Holding N locks across an atomic block is a real contention amplifier — exactly what the per-file-lock rebuild exists to avoid for ordinary writes. An atomic block over 100 files holds 100 advisory locks; any other writer wanting any of those waits. "Use judiciously" is concrete advice.
- `atomic` and `version_token` solve adjacent but distinct problems. The token is *detect-and-retry* — other writers can interleave; you find out at write time and retry. Atomic is *prevent-interleave* — no other writer touches the locked files for the duration. Macro authors reach for atomic when the read-write sequence is inherently transactional (multi-file, complex logic, retry semantics awkward). For single read-write pairs `version_token` is usually the better choice.

*One subtler design call the future implementer must make.* The lock set the engine pre-acquires covers the *write destinations* of the atomic block — reads inside the block stay lock-free per INV-07. An `atomic { $a = get_document(A); write_document(B, transform($a)); }` block locks `B` (plus its ancestor shared directory locks); the read of `A` is not under the atomic block's lock set, and another writer can modify A in parallel with the block. If the macro author needs the read of A to be consistent with the write of B, they must add A to the atomic intent — either by listing it explicitly (e.g., a `reads: [A]` argument to the block) or by accepting the read is racy and using `version_token` on A through the write. The language design should pin this down; keeping the locks write-only honors INV-07 and matches how the rest of the system models reads.

## 4. Implications for FlashQuery

The primary implication is architectural: the locking layer should be a proper subsystem with a stable interface, not an implementation detail scattered across tool handlers. The four hypotheses from the original capture were tested against the codebase during research (2026-05-24):

- **"Per-file locks with a lock registry may be the right default."** — **Confirmed.** This is the recommended direction (§3.7). It does not require new infrastructure: the lock mechanism already accepts arbitrary keys, and `files.ts` already uses a fine-grained key. Implement the registry with lock striping to bound memory, and acquire multi-file locks in sorted-path order to avoid deadlock.
- **"Write-to-temp + atomic rename for document writes."** — **Already implemented**, not a future idea: `vaultManager.writeMarkdown` does exactly this (`vault.ts:214-216`). The outstanding work is durability (`fsync` of file and directory) and a unique temp name, not the rename itself (§3.5).
- **"SQLite WAL mode … used already for the identity/tracking layer."** — **Factually incorrect.** FlashQuery has no SQLite. The identity/tracking layer is **Postgres via Supabase** (the `fqc_*` tables); Supabase is a hard dependency. The WAL *concept* — single writer per object plus snapshot reads — is still a useful mental model, and FlashQuery already gets the snapshot-read half from atomic rename. But there is no SQLite WAL to lean on, and the metadata side is coordinated (today) by the `fqc_write_locks` table, whose defect is described in §3.4. This hypothesis should be struck from the design assumptions.
- **"Cooperative chunking for long operations."** — **Confirmed** as the right pattern for the scanner and reconciliation passes (§3.7 step 5, OQ6).

The interface implication needs one correction. The original goal — "change what happens inside `acquireLock(resource)` / `releaseLock(resource)` without changing the call sites" — is **not fully achievable**. Per-file granularity requires the lock *key* to carry the file path, and only the caller knows the path. The signature of `acquireLock`/`releaseLock` does not change, but each call site must construct a per-file key instead of passing a literal. This is a small, mechanical change (best hidden behind a `withDocumentLock` helper), not a zero-change drop-in.
