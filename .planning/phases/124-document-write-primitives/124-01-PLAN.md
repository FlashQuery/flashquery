<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client -> write_document schema | Untrusted mode/path/frontmatter/tags enter server write code. |
| write_document -> vault filesystem | User-provided paths become vault-relative filesystem writes. |
| write_document -> Supabase | Handler updates `fqc_documents` metadata and embeddings asynchronously. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-124-01 | Tampering | `write_document(mode:"create")` path | mitigate | Reuse vault-root relative path guard; path traversal returns canonical `invalid_input` or `conflict`, never writes outside vault. |
| T-124-02 | Tampering/Spoofing | reserved frontmatter | mitigate | Reject managed `FM.*` fields before write using `Object.values(FM)` and return `invalid_input` with `details.field`. |
| T-124-03 | Tampering | update identifier resolution | mitigate | Resolve exactly one document with `resolveDocumentIdentifier`; propagate `ambiguous_identifier`/`not_found` via `jsonExpectedError`. |
| T-124-04 | Denial of Service/Race | document write lock | mitigate | Preserve existing document lock and map lock contention to canonical `conflict` with `details.reason:"lock_contention"` when locking is enabled. |
| T-124-05 | Information Disclosure | disabled-category leakage in apply_tags | transfer | Covered in Plan 04 where `apply_tags` is implemented; no memory target handling in this plan. |
</threat_model>
