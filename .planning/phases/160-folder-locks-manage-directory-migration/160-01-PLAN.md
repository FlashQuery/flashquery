<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP path input to lock service | File and directory paths originating from tool calls become canonical directory lock resources. |
| Filesystem to canonicalization | Symlinks, missing descendants, and path traversal attempts affect which directories are locked. |
| Lock service to PostgreSQL | Canonical `dir:` resources become parameterized bigint advisory-lock keys. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-160-01 | Tampering | `withAncestorDirectoryLocksShared` | mitigate | Enumerate only canonical ancestors from file parent through vault root and never lock paths outside `config.instance.vault.path`. |
| T-160-02 | Denial of Service | shared/exclusive advisory acquisition | mitigate | Use Phase 159 bounded `pg_try_advisory_lock_shared` / `pg_try_advisory_lock` retry loops and raise `LockTimeoutError` instead of indefinite waits. |
| T-160-03 | Tampering | PostgreSQL advisory SQL | mitigate | Use parameterized `$1::bigint` SQL for `pg_try_advisory_lock_shared`, `pg_advisory_unlock_shared`, `pg_try_advisory_lock`, and `pg_advisory_unlock`. |
| T-160-04 | Elevation of Privilege | public lock service exports | mitigate | `tests/unit/lock-helper-only.test.ts` allows only high-level facades and blocks raw acquire/release primitive exports. |
| T-160-SC | Tampering | npm/pip/cargo installs | accept | No package install is planned; existing `pg`, `async-mutex`, and Vitest dependencies are reused. |
</threat_model>
