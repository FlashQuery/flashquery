<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client -> write_record validator | Untrusted parameters request dynamic plugin-table writes. |
| Plugin schema -> record validation | Plugin-authored schemas define accepted user fields and required fields. |
| Handler -> Supabase dynamic table | Validated data crosses into persistent plugin tables. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-126-01 | Tampering | `src/mcp/utils/record-validation.ts` | mitigate | Reject generated fields including `id`, timestamps, `instance_id`, `fqc_id`, `path`, `status`, and embedding columns before DB mutation. |
| T-126-02 | Tampering | `write_record` data validation | mitigate | Reject unknown fields not declared in `PluginTableSpec.columns`; return `invalid_input` with field details and `isError:false`. |
| T-126-03 | Elevation of privilege | Dynamic table writes | mitigate | Require explicit `mode`, `plugin_id`, and `table`; no inferred create/update behavior. |
| T-126-04 | Information disclosure | Record output helpers | mitigate | Gate full `data` and `schema_metadata` behind explicit include values for write responses. |
| T-126-05 | Repudiation | Traceability ledger | mitigate | Create requirement-to-evidence table before implementation so validation evidence cannot drift silently. |
</threat_model>
