<interfaces>
From `src/services/document-lock.ts`:
- `withDocumentLock(config, filePath, fn)` canonicalizes an existing or absent destination path and runs `fn` under one per-file lock.
- `withDocumentLocks(config, filePaths, fn)` derives unique canonical `file:` entries, sorts by `basicKey`, acquires advisory locks in that order, and releases in reverse order.
- `withAncestorDirectoryLocksShared(config, filePath, fn)` wraps file writes with shared parent directory locks from Phase 160.
- `__testing.deriveDocumentLockEntry(config, resourcePath, kind?)` is the allowed test boundary for deterministic canonical key/order assertions.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client to document tools | Untrusted destination paths enter write/copy/move handlers. |
| Document tools to filesystem | Validated paths become filesystem reads, writes, renames, and existence checks. |
| Document tools to advisory-lock facade | Canonical path inputs become advisory-lock keys. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-161-01 | Tampering | write/copy/move destination paths | mitigate | Keep `validateVaultPath` before locks and assert authoritative `existsSync` checks run inside destination file locks. |
| T-161-02 | Denial of Service | move source+destination multi-lock | mitigate | Use existing bounded `withDocumentLocks` timeout and sorted canonical ordering per INV-09 to avoid deadlocks. |
| T-161-SC | Tampering | package installs | accept | No package installs are planned for this phase. |
</threat_model>
