**Test 1** — `it('50 new docs auto-tracked in single reconciliation pass')`:
- Use `Promise.all` to write 50 `.md` files to the `docs/` folder
- Invalidate reconciliation cache (`invalidateReconciliationCache()` imported from plugin-reconciliation.ts)
- Call `search_records`
- Assert response text contains "Auto-tracked 50 new document(s)"
- Query plugin table — assert 50 rows with `status = 'active'`

**Test 2** — `it('count-based summary — no item enumeration')`:
- Write 3 new `.md` files to docs folder
- Call record tool
- Assert response text matches `Auto-tracked 3 new document(s)` (count only, no file names)
- Assert response text does NOT contain any `.md` file name or path from the test files

**Test 3** — `it('intermediate states invisible during bulk processing')`:
- Write 10 `.md` files, call record tool (all tracked)
- Delete 3 files, move 2 files (rename in same folder)
- Call record tool again (triggers reconciliation for modified/deleted)
- Assert the response is well-formed and no errors thrown
- Assert deleted files result in archived plugin rows

**Test 4** — `it('no spurious modified classification after auto-track completes')`:
- Write 5 `.md` files, call record tool (auto-tracks all, sets last_seen_updated_at)
- Immediately call record tool again (within staleness window — or invalidate cache first to force re-run)
- Assert response does NOT contain "Synced fields on N modified" — because auto-track correctly set `last_seen_updated_at` to prevent false modified classification (RECON-05)

**Test 5** — `it('incremental pending review processing — no duplicates across rounds')`:
- Use a plugin with `template` declared (create a second plugin YAML or use a type with template)
- Auto-track 3 docs (creates 3 pending review rows)
- Call `clear_pending_reviews` in query mode — get the fqc_ids
- Clear 1 row
- Auto-track 2 more docs (2 more pending review rows)
- Call query mode — assert 4 total (not 5, because 1 was cleared; and no duplicates from re-reconciliation)

**Test 6** — `it('access: read-only guardrail emits warning in create_document response')` (RO-60):
- Register a document tools handler via mock server (`registerDocumentTools`)
- Call `create_document` with a path inside `readonly-folder/` (the read-only plugin folder)
- Assert response text contains "Warning: Plugin" and "read-only access"
- Assert `result.isError` is falsy (write proceeds despite warning)
- Assert the file was actually created in the vault (write did proceed)
