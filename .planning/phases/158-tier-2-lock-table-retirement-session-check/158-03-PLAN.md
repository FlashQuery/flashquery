<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| startup -> Postgres sessions | Startup validates cross-session advisory-lock visibility before serving MCP. |
| database URL config -> operator guidance | Misconfigured external database endpoints must fail without leaking secrets. |
| docs -> operator setup | Documentation must not steer users toward transaction-mode poolers or manual unlock workflows. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-158-08 | Tampering | `lock-startup.ts` advisory SQL | mitigate | Use parameterized `$1::bigint` queries for acquire, observe, and release. |
| T-158-09 | Denial of Service | startup self-test probe lock | mitigate | Use a deterministic throwaway key, release in `finally`, check unlock result, and avoid long-lived transactions. |
| T-158-10 | Spoofing | transaction-pooler silent misconfiguration | mitigate | Observe session advisory lock from a second checkout through `pg_locks`; fail startup if not visible. |
| T-158-11 | Information Disclosure | startup failure message | mitigate | Name suspected transaction-mode pooler and docs, but do not print raw `DATABASE_URL` or credentials. |
| T-158-SC | Tampering | npm/pip/cargo installs | accept | No package installs are planned; implementation uses existing `pg` and `withPgClient`. |
</threat_model>
