<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP/tool caller -> storage helper | Callers must pass already validated absolute vault paths; this primitive must not become a raw user-path API. |
| Storage helper -> filesystem | Filesystem operations can fail or partially complete; errors must surface and visible destination must be committed only by rename. |
| Test injection -> production module | Test hooks must not let production callers bypass durable behavior. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-156-01 | Tampering | `writeVaultFile` commit sequence | mitigate | Write only to same-directory unique temp files and commit with `rename`; never write destination bytes directly. |
| T-156-02 | Repudiation | filesystem failure handling | mitigate | Propagate write, fsync, rename, and directory fsync errors; unit tests simulate each failure class. |
| T-156-03 | Denial of Service | temp files after failed writes | mitigate | Best-effort temp cleanup on error and Phase 156 cleanup migration in later plans. |
| T-156-04 | Elevation of Privilege | absolute path input | mitigate | Keep path validation in existing callers; do not expose `writeVaultFile` as an MCP tool or accept raw user paths. |
| T-156-SC | Tampering | npm installs/native adapter | mitigate | No package install by default; blocking human checkpoint and legitimacy audit required for any native F_FULLFSYNC adapter. |
</threat_model>
