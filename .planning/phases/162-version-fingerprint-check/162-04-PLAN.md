<objective>
Implement expected-version checks and success tokens for top-level document write tools.

Purpose: Give callers opt-in read-to-write conflict detection without changing existing last-writer-wins behavior for callers that omit `expected_version` / `if_match`.

</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP caller -> write tool | Untrusted version preconditions enter document mutation paths. |
| Locked filesystem bytes -> write decision | Fresh raw disk bytes determine whether a write may proceed. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-162-10 | Tampering | Top-level document writes | mitigate | Compare `expected_version` against fresh raw bytes inside `withDocumentLock` / `withDocumentLocks`; refuse before write on mismatch. |
| T-162-11 | Information Disclosure | Top-level conflict envelopes | mitigate | Structural/destructive tools return whole document only where REQ-015 requires it; no unrelated DB metadata is included. |
| T-162-12 | Elevation of Privilege | Version-check path resolution | mitigate | Keep existing resolver and `validateVaultPath` call order; checks consume already-resolved locked file paths. |
| T-162-SC | Tampering | npm installs | accept | No package installs are planned. |
</threat_model>
